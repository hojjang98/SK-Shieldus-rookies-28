## 2. 실습 내용 정리

### 실습 1: Chrome 개발자 도구를 이용한 웹 페이지 분석

**목표:** 웹 페이지의 HTML 요소를 검사하고 실시간으로 데이터를 수정하여 클라이언트 사이드 검증의 한계 이해

**실습 환경:**
- Google Chrome 브라우저
- 한빛출판네트워크 웹사이트 (hanbit.co.kr)
- 개발자 도구 (F12 또는 Ctrl+Shift+I)

**실습 단계:**
```bash
# 1. Chrome 브라우저에서 한빛출판네트워크 접속
# URL: https://www.hanbit.co.kr

# 2. 개발자 도구 실행
# [설정] - [도구 더보기] - [개발자 도구] 클릭 또는 F12

# 3. Element Inspector 활성화
# 개발자 도구 좌측 상단 <Select an element> 아이콘 클릭

# 4. 웹 페이지 특정 영역 선택
# 수정하고자 하는 텍스트나 이미지 영역 클릭

# 5. HTML 코드 실시간 수정
# Elements 탭에서 텍스트 더블클릭하여 내용 변경
# 예: 도서 제목, 가격 정보 등 수정
```

**분석 포인트:**
```
확인 항목:
✓ HTML 구조 및 CSS 스타일 분석
✓ 클라이언트 사이드에서 수정 가능한 데이터 식별
✓ hidden 필드, disabled 속성 등 제약 조건 확인
✓ JavaScript 이벤트 핸들러 및 검증 로직 파악
```

**보안 인사이트:**
- 클라이언트 사이드 검증만으로는 보안 확보 불가능
- 중요한 비즈니스 로직은 반드시 서버 사이드에서 재검증 필요
- 가격, 수량, 권한 등 민감한 데이터는 클라이언트에서 조작 가능
- 개발자 도구는 공격자의 기본 도구이며, 일반 사용자도 쉽게 접근 가능

### 실습 2: Burp Suite의 Target 기능으로 웹 사이트 구조 분석

**목표:** Burp Suite를 통해 웹 애플리케이션의 디렉터리 구조와 파일 목록을 시각화하고 공격 표면 파악

**실습 환경:**
- Burp Suite Community Edition
- 웹 브라우저 (프록시 설정)
- 한빛출판네트워크 웹사이트

**실습 단계:**
```bash
# 1. Burp Suite 실행 및 프록시 설정
# Burp Suite 실행 → [Proxy] 탭 → [Intercept is on] 확인

# 2. 브라우저 프록시 설정
# 브라우저 설정 → 프록시: localhost:8080

# 3. 웹 사이트 탐색
# 한빛출판네트워크의 다양한 메뉴 클릭 (홈, 도서, 이벤트 등)
# 로그인 시도 (있는 경우)

# 4. Intercept 비활성화
# [Proxy] 탭 → <Intercept is on> 클릭 → <Intercept is off>

# 5. Target 탭에서 사이트 구조 확인
# [Target] 탭 → Site map → 도메인 확장하여 디렉터리 구조 확인
# 각 리소스 클릭 시 Request/Response 상세 정보 표시
```

**분석 포인트:**
```
사이트 구조 분석:
✓ 디렉터리 계층 구조 (academy, brand, channel, event 등)
✓ 파일 확장자별 분류 (html, js, css, 이미지 등)
✓ 파라미터 전달 방식 (GET/POST)
✓ 쿠키 및 세션 관리 방식
```

**Target 기능 활용:**
- 웹 애플리케이션의 전체 공격 표면 시각화
- 숨겨진 디렉터리나 파일 발견 가능성
- 각 엔드포인트별 파라미터 및 응답 코드 확인
- 이후 Intruder나 Repeater 공격 대상 선정에 활용

**보안 고려사항:**
```
탐지 패턴:
- 비정상적으로 많은 페이지 요청
- 짧은 시간 내 다양한 URL 패턴 접근
- robots.txt 무시한 크롤링 시도

방어 방법:
→ Rate Limiting 구현 (요청 빈도 제한)
→ 디렉터리 리스팅 비활성화
→ 민감한 파일/디렉터리 접근 제어
```

### 실습 3: Burp Suite Intruder로 자동화 공격 시뮬레이션

**목표:** 변숫값을 자동으로 증가시키며 여러 페이지를 탐색하는 Fuzzing 공격 수행

**실습 환경:**
- Burp Suite Community Edition
- 한빛출판네트워크 이벤트 페이지
- Target URL: /event/current/current_event_view.html?hbe_idx=96

**실습 단계:**
```bash
# 1. Target에서 공격 대상 선택
# [Target] 탭 → current_event_view.html (hbe_idx=96&page=0)
# 우클릭 → [Send to Intruder]

# 2. Intruder Positions 설정
# [Intruder] → [Positions] 탭
# hbe_idx 파라미터 값($96$)이 자동으로 페이로드 포지션 지정됨
# Attack type: Sniper (단일 변수 공격)

# 3. Payloads 설정
# [Payloads] 탭 클릭
# Payload type: Numbers 선택
# From: 96
# To: 105
# Step: 1

# 4. 공격 실행
# <Start attack> 버튼 클릭

# 5. 결과 분석
# Status, Length 컬럼 확인
# Status 200: 페이지 존재
# Status 404/500: 페이지 없음 또는 오류
# Length 차이로 컨텐츠 유무 판단
```

**Intruder 공격 타입:**
```
공격 방식:
1. Sniper: 단일 변수 순차 공격
2. Battering ram: 모든 포지션에 동일 값 대입
3. Pitchfork: 각 포지션마다 다른 페이로드 세트 매칭
4. Cluster bomb: 모든 조합 테스트
```

**발견 가능한 취약점:**
- 권한 없는 리소스 접근 (IDOR)
- 숨겨진 페이지나 관리자 페이지 노출
- 예측 가능한 세션 ID 패턴
- 순차적 ID 기반의 정보 열람

**보안 고려사항:**
```
탐지 패턴:
- 동일 IP에서 짧은 시간 내 대량 요청
- 순차적/패턴화된 파라미터 값
- 비정상적인 User-Agent 또는 없는 Referer

방어 방법:
→ CAPTCHA 또는 Rate Limiting 적용
→ 세션 기반 접근 제어 및 권한 검증
→ UUID 등 예측 불가능한 식별자 사용
```

### 실습 4: Burp Suite Repeater로 수동 취약점 검증

**목표:** 특정 요청을 반복 전송하며 파라미터를 수동으로 조작하여 서버 응답 분석

**실습 환경:**
- Burp Suite Community Edition
- 한빛출판네트워크 이벤트 상세 페이지

**실습 단계:**
```bash
# 1. Target에서 요청 선택
# [Target] 탭 → 특정 페이지 우클릭 → [Send to Repeater]

# 2. Repeater 탭으로 이동
# [Repeater] 탭 클릭
# Request 창에 HTTP 요청 내용 표시

# 3. 파라미터 수동 수정
# Request 창에서 GET 파라미터 또는 POST 데이터 수정
# 예: hbe_idx=96 → hbe_idx=97

# 4. 요청 전송 및 응답 확인
# <Send> 버튼 클릭
# Response 창에서 HTTP 상태 코드, 헤더, 본문 확인

# 5. 다양한 입력값 테스트
# SQL Injection: ' OR '1'='1
# XSS: <script>alert(1)</script>
# Path Traversal: ../../etc/passwd
```

**Repeater 활용 시나리오:**
```
테스트 케이스:
1. 파라미터 타입 변경 (숫자 → 문자열)
2. 특수문자 삽입 (', ", <, >, 등)
3. 범위 초과 값 입력 (음수, 매우 큰 수)
4. NULL 또는 빈 값 전송
```

**보안 인사이트:**
- 서버 사이드 입력 검증 강도 확인
- 에러 메시지를 통한 정보 노출 여부 파악
- 인증/인가 우회 가능성 테스트
- 비즈니스 로직 취약점 발견

### Burp Suite 공격 타입 상세 분석

| 구분 | Sniper | Battering Ram | Pitchfork | Cluster Bomb | 분석/인사이트 |

### 피싱 공격 시나리오 분석

```python
# 피싱 공격 흐름 예시

# 1단계: 미끼 메시지 작성
phishing_message = """
[긴급] 페이스북 계정 보안 경고

고객님의 계정에서 의심스러운 로그인이 감지되었습니다.
24시간 내 본인 확인을 하지 않으면 계정이 정지될 수 있습니다.

아래 링크를 클릭하여 본인 확인을 진행해주세요:
http://faceb00k-security.com/verify  # 오타로 위장
"""

# 2단계: 가짜 로그인 페이지 생성
# - 진짜 페이스북과 거의 동일한 디자인
# - URL은 유사하지만 다른 도메인 (faceb00k, facebok 등)
# - HTTPS 인증서도 무료로 발급 가능 (Let's Encrypt)

# 3단계: 정보 탈취
stolen_credentials = {
    "email": user_input_email,
    "password": user_input_password,
    "ip_address": request.remote_addr,
    "user_agent": request.headers.get('User-Agent')
}

# 4단계: 정상 페이스북으로 리다이렉트
# - 사용자는 "로그인 실패" 정도로 인식
# - 공격자는 이미 자격증명 획득 완료
```

**피싱 탐지 및 방어:**
```
탐지 방법:
- URL 철자 확인 (페이스북 → faceb00k 등)
- HTTPS 인증서 발급자 확인 (신뢰할 수 있는 CA인지)
- 로그인 전 주소창 재확인
- 이메일 발신자 도메인 검증

방어 방법:
- 안티피싱 브라우저 확장 프로그램 사용
- 이메일 내 링크 클릭 금지, 직접 사이트 접속
- 2FA(Two-Factor Authentication) 활성화
- 정기적인 피싱 교육 실시
```

| **스캔 단계** | - 짧은 시간 내 다수 URL 접근<br>- 404 에러 대량 발생<br>- 알려진 스캐너 User-Agent | `192.168.1.100 - - [23/Dec/2025:14:32:11] "GET /admin HTTP/1.1" 404` | - IP 기반 Rate Limiting<br>- 스캐너 시그니처 차단<br>- 허니팟 디렉터리 설정 |
| **SQLi 시도** | - SQL 문법 키워드 탐지<br>- 특수문자 다수 포함<br>- 응답 시간 지연 (Blind SQLi) | `id=1' OR '1'='1 -- ` | - WAF 시그니처 업데이트<br>- 파라미터 화이트리스트<br>- DB 에러 메시지 숨김 |
| **브루트 포스** | - 동일 계정 다수 로그인 실패<br>- 순차적 ID/PW 시도<br>- 높은 요청 빈도 | `Failed login for user 'admin' from 10.0.0.50 (attempt 15/20)` | - 계정 잠금 정책 적용<br>- CAPTCHA 도입<br>- IP 차단 및 알림 |

### 웹 방화벽 (WAF) 룰셋 예시

```bash
# ModSecurity 기반 WAF 룰 예시

# 1. SQL Injection 방어
SecRule ARGS "@rx (?i)(union|select|insert|update|delete|drop|create)" \
    "id:1001,phase:2,deny,status:403,msg:'SQL Injection Detected'"

# 2. XSS 방어
SecRule ARGS "@rx <script[^>]*>.*?</script>" \
    "id:1002,phase:2,deny,status:403,msg:'XSS Attack Detected'"

# 3. Path Traversal 방어
SecRule ARGS "@rx \.\./|\.\.\\" \
    "id:1003,phase:2,deny,status:403,msg:'Path Traversal Detected'"

# 4. 알려진 스캐너 차단
SecRule REQUEST_HEADERS:User-Agent "@rx (nikto|nmap|sqlmap|burp)" \
    "id:1004,phase:1,deny,status:403,msg:'Known Scanner Detected'"

# 5. Rate Limiting
SecAction "id:1005,phase:1,nolog,pass,initcol:ip=%{REMOTE_ADDR}"
SecRule IP:REQUEST_COUNT "@gt 100" \
    "id:1006,phase:1,deny,status:429,msg:'Rate Limit Exceeded'"
```

## 6. 배운 점 및 인사이트

### 새로 알게 된 점

- **웹 브라우저 개발자 도구의 강력함:** 단순한 디버깅 도구가 아닌 보안 테스트의 출발점. 클라이언트 사이드 검증만으로는 보안을 담보할 수 없다는 것을 실습으로 확인. 개발자 도구는 모든 사용자가 접근 가능하므로 중요한 로직은 반드시 서버 사이드에서 처리해야 함.

- **Burp Suite의 다양한 활용 방안:** Target으로 사이트 구조 파악, Intruder로 자동화 공격, Repeater로 정밀 분석. 하나의 도구로 정찰부터 취약점 검증까지 전 과정을 커버할 수 있음. Community 버전도 충분히 강력하며, Pro 버전은 자동 스캐닝 추가.

- **Acunetix의 상용 스캐너 품질:** 자동 크롤링, 휴리스틱 분석, 다양한 보고서 템플릿. 기술팀, 경영진, 감사팀 등 대상별 맞춤 보고서 생성 가능. 오탐률이 낮고 재현 방법까지 상세히 제공하여 실무 활용도가 높음.

- **SNS 보안 위협의 현실성:** 웹 2.0 환경에서 정보가 기하급수적으로 증가하면서 개인정보, 기업 기밀, 국가 정보까지 SNS를 통해 유출. 이블 트윈 어택은 시스템적 차단이 불가능하며, 사용자 경각심이 유일한 방어책.

- **통합적 보안 접근의 필요성:** 도구만으로는 부족하며, 개발 보안, 네트워크 보안, 사용자 교육이 모두 결합되어야 효과적. SOC 분석가는 웹 공격 패턴을 이해하고 WAF, SIEM과 연계한 탐지 체계를 구축해야 함.

### 이전 학습과의 연결고리

- **XSS/CSRF → 브라우저 개발자 도구 활용:** 이전에 학습한 XSS와 CSRF 취약점을 개발자 도구로 재현하고 분석할 수 있음. JavaScript 코드 삽입, 쿠키 탈취 시뮬레이션 등 실습 확장 가능.

- **SQL Injection → Burp Suite/Sqlmap 연계:** 수동으로 학습한 SQL Injection을 Burp Suite Repeater로 정밀 테스트하고, Sqlmap으로 자동화 공격 수행. 각 도구의 장단점 비교를 통해 상황별 최적 도구 선택 가능.

- **소스코드 분석 → 웹 취약점 스캐너 결과 해석:** 취약점 스캐너가 발견한 결과를 소스코드 수준에서 검증하는 능력. Acunetix가 제시한 취약점이 실제로 존재하는지, False Positive인지 판단하는 기준 확립.

- **네트워크 보안 → 웹 프록시 트래픽 분석:** Wireshark로 학습한 패킷 분석 지식을 HTTP/HTTPS 트래픽에 적용. Burp Suite는 Application Layer 프록시로서 Wireshark의 상위 레이어 분석 도구.

### 실무 적용 아이디어

**SOC 분석가 관점:**

- **웹 공격 시그니처 데이터베이스 구축:** Burp Suite로 발견한 공격 패턴을 SIEM 룰로 변환. SQL Injection, XSS 등의 페이로드를 시그니처화하여 실시간 탐지 체계 구축.

- **웹 취약점 스캔 결과 상관 분석:** Acunetix 스캔 결과와 실제 공격 로그를 매칭하여 우선순위 산정. 스캐너가 발견한 High 등급 취약점이 실제 공격당하고 있는지 확인.

- **SNS 위협 인텔리전스 수집:** 기업명, 임원 이름으로 SNS 모니터링 자동화. 가짜 계정, 피싱 페이지, 정보 유출 징후를 조기에 발견하고 대응.

**침투 테스트 관점:**

- **모의 해킹 시나리오 개발:** Burp Suite의 다양한 기능을 활용하여 실전과 유사한 침투 테스트 수행. Target → Intruder → Repeater 순서로 체계적인 공격 수행.

- **취약점 검증 자동화:** Sqlmap, Acunetix 등 스캐너 결과를 수동 검증하는 프로세스 정립. 오탐 제거 및 실제 익스플로잇 가능 여부 확인.

**웹 개발자 관점:**

- **Secure Coding 체크리스트 작성:** Burp Suite와 Acunetix에서 발견되는 취약점 패턴을 기반으로 개발 단계 보안 점검 항목 생성. 코드 리뷰 시 활용.

- **개발자 도구 활용 교육:** 개발팀 내부에서 브라우저 개발자 도구 활용법 교육. 클라이언트 사이드 검증의 한계를 인식하고 서버 사이드 검증 강화.

웹 보안 도구는 단순한 소프트웨어가 아닌 보안 사고방식을 체화하는 학습 도구다. 브라우저 개발자 도구로 클라이언트 검증의 한계를 직접 확인하고, Burp Suite로 HTTP 트래픽의 모든 측면을 분석하며, Acunetix로 자동화 스캔의 효율성을 경험했다. 이러한 도구들은 공격자와 방어자 모두가 사용하며, 차이는 목적과 윤리에 있다. SOC 분석가는 공격자의 도구와 기법을 이해해야만 효과적인 탐지 룰을 작성할 수 있고, 개발자는 이러한 도구로 자신의 코드가 어떻게 공격받을 수 있는지 미리 체험해야 한다. SNS 보안 위협은 기술적 취약점을 넘어 인간의 심리와 신뢰를 악용하며, 시스템적 방어가 불가능한 영역도 존재한다는 점에서 교육과 경각심이 최후의 방어선임을 깨달았다. 앞으로는 이러한 도구들을 실무 환경에 적용하여 웹 애플리케이션 보안 진단, 침투 테스트, SOC 탐지 룰 개발에 활용할 것이며, 특히 Burp Suite와 웹 취약점 스캐너를 결합한 체계적인 진단 프로세스를 확립하는 것이 다음 학습 목표다.
