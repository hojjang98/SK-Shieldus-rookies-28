# Day 38: 침해사고 대응 실무

## 1. 침해사고 대응 개요

### 1.1 침해사고란?

**정의:**
정보시스템 또는 개인정보에 대한 **불법적인 접근, 유출, 훼손, 파괴** 등의 사고

**유형:**
- 해킹 (외부 침입)
- 개인정보 유출
- 랜섬웨어 감염
- DDoS 공격
- 내부자 유출
- 악성코드 감염
- 시스템 장애 (보안 사고 연계)

### 1.2 왜 대응 절차가 중요한가?

#### 법적 의무

**개인정보보호법 제34조 (개인정보 유출 통지 등):**
개인정보 유출 시 **즉시** 통지 의무

**정보통신망법 제27조의3 (개인정보의 유출 통지 등):**
동일

**지연 시:**
- 과태료
- 과징금
- 형사 처벌

#### 피해 최소화

**신속한 대응:**
- 추가 유출 차단
- 2차 피해 방지

**늦은 대응:**
- 피해 확대
- 복구 비용 증가

#### 신뢰도

**적절한 대응:**
- 고객 신뢰 유지

**미흡한 대응:**
- 평판 추락
- 고객 이탈

### 1.3 침해사고 대응 생명주기
```
1. 준비 (사전)
   ↓
2. 탐지 및 분석
   ↓
3. 억제 (Containment)
   ↓
4. 제거 (Eradication)
   ↓
5. 복구 (Recovery)
   ↓
6. 사후 조치 (Post-Incident)
```

**출처:**
NIST Cybersecurity Framework

## 2. 1단계: 준비 (사전)

### 2.1 침해사고 대응팀 구성

**CSIRT (Computer Security Incident Response Team)**

#### 팀 구성

**필수 인원:**

**1. 총괄 책임자 (CISO)**
- 역할: 전체 대응 지휘
- 의사 결정

**2. 기술팀 리더**
- 역할: 기술적 대응 총괄

**3. 보안 엔지니어 (2~3명)**
- 역할: 로그 분석, 침입 경로 파악, 조치

**4. 시스템 관리자 (1~2명)**
- 역할: 시스템 복구, 패치

**5. 네트워크 관리자 (1명)**
- 역할: 네트워크 차단, 트래픽 분석

**6. 법무팀 (1명)**
- 역할: 법적 대응, 신고, 고소

**7. 홍보팀 (1명)**
- 역할: 대외 커뮤니케이션

**8. 경영진 연락 담당 (1명)**
- 역할: 경영진 보고

#### 연락망

**24시간 연락 가능:**
```

보안 엔지니어:
- 이영희: 010-3456-7890

...
```

### 2.2 침해사고 대응 절차 수립

#### 대응 절차서 작성

**목차:**
```
1. 목적
2. 적용 범위
3. 용어 정의
4. 침해사고 정의 및 분류
5. 대응 조직
6. 대응 절차
   6.1 탐지
   6.2 신고
   6.3 초동 대응
   6.4 분석
   6.5 복구
   6.6 사후 조치
7. 신고 체계
8. 외부 기관 연락처
9. 첨부
   - 침해사고 신고서 양식
   - 체크리스트
```

#### 침해사고 분류 및 등급

**등급:**

**Level 1 (Critical)**
- 대규모 개인정보 유출 (1만명 이상)
- 주요 시스템 마비
- 금전적 피해 발생

**대응:**
- CSIRT 전체 소집
- 경영진 즉시 보고
- 외부 전문가 투입

**Level 2 (High)**
- 중규모 개인정보 유출 (1천명~1만명)
- 일부 시스템 마비
- 서비스 일부 중단

**대응:**
- CSIRT 주요 인원 소집
- 경영진 보고

**Level 3 (Medium)**
- 소규모 개인정보 유출 (1천명 미만)
- 시스템 장애 (경미)

**대응:**
- 보안팀 대응
- CISO 보고

**Level 4 (Low)**
- 악성코드 탐지 (확산 전)
- 보안 정책 위반

**대응:**
- 담당자 대응

#### 보안 시스템

**필수:**
- [ ] 침입탐지시스템 (IDS)
- [ ] 침입차단시스템 (IPS)

- [ ] 메모리 덤프 도구
- [ ] 로그 분석 도구
- [ ] 네트워크 패킷 분석 도구 (Wireshark 등)

#### 백업

**정기 백업:**
- 일일 백업
- 주간 전체 백업
- 백업본 별도 보관

**백업 테스트:**
- 분기 1회 복구 테스트

**모의 훈련:**
연 1회 이상

**시나리오:**
```

```

**평가:**
- 대응 시간
- 의사 결정
- 커뮤니케이션

## 3. 2단계: 탐지 및 분석

### 3.1 탐지

#### 탐지 방법

**(1) 자동 탐지**
- IDS/IPS 알림
- SIEM 알림
- 백신 탐지

**(2) 모니터링**
- 보안 관제 센터 (SOC)
- 로그 점검

**(3) 신고**
- 직원 신고
- 고객 신고
- 외부 기관 통보 (KISA 등)

#### 초기 징후

**의심 징후:**
- [ ] 비정상 로그인 시도 (다량)
- [ ] 비정상 트래픽 (급증)
- [ ] 시스템 느려짐
- [ ] 파일 변조 탐지
- [ ] 백신 알림
- [ ] 외부 기관 통보
- [ ] 다크웹 개인정보 발견 신고

### 3.2 신고 및 보고

#### 내부 신고

**발견자 → 보안팀:**
즉시 신고

**신고 채널:**
- 비상 연락망 (전화)
- 이메일
- 메신저

**신고 내용:**
```
[침해사고 신고]

신고 일시: 2024년 4월 1일 15:30
신고자: 홍길동 (시스템 관리자)
발견 경위: 서버 로그 점검 중 비정상 접근 발견
이상 징후:
- 서버 로그에 unknown IP에서 대량 접근
- IP: 123.456.789.000
- 시간: 14:00 ~ 15:00
- 접근 시도 횟수: 10,000회
```

#### 경영진 보고

**CISO → 경영진:**

**Level 1 (Critical):**
즉시 보고 (30분 이내)

**Level 2 (High):**
1시간 이내

**보고 내용:**
```
[긴급] 침해사고 발생 보고

발생 일시: 2024.4.1 15:30
탐지 일시: 2024.4.1 15:35
사고 등급: Level 1 (Critical)

사고 개요:
- 외부 IP에서 웹서버 해킹 시도
- SQL Injection 공격
- 고객 DB 접근 흔적 발견

예상 피해:
- 고객 개인정보 유출 가능성
- 규모: 조사 중 (최대 10만건 추정)

초동 조치:
- 해당 서버 네트워크 차단
- 침입 경로 조사 중
- CSIRT 소집 완료

향후 조치:
- 유출 범위 확인
- 법적 신고 준비
- 고객 통지 준비

보고자: 김철수 (CISO)
```

### 3.3 초동 대응

**우선 순위:**
1. **추가 피해 차단**
2. 증거 보존
3. 원인 파악

#### 즉시 조치

**(1) 격리 (Isolation)**
```
침해당한 시스템을 네트워크에서 격리

방법:
- 네트워크 케이블 제거
- 방화벽 차단
- 계정 비활성화

주의:
- 전원은 끄지 말 것 (메모리 증거 소실)
```

**(2) 계정 차단**
```
의심 계정 즉시 비활성화

- 해킹당한 계정
- 비정상 활동 계정
```

**(3) 비밀번호 변경**
```
주요 계정 비밀번호 즉시 변경

- 관리자 계정
- 시스템 계정
```

**(4) 백업 확인**
```
최신 백업 확보 여부 확인

- 언제 백업?
- 정상 작동?
- 복구 가능?
```

#### 증거 보존

**(1) 로그 보존**
```
모든 로그 즉시 백업

- 시스템 로그
- 애플리케이션 로그
- 네트워크 로그
- 방화벽 로그
- 웹 로그
```

**(2) 메모리 덤프**
```
침해당한 서버 메모리 덤프

도구:
- FTK Imager
- Volatility
```

**(3) 디스크 이미징**
```
침해당한 서버 디스크 이미징

도구:
- dd
- FTK Imager
```

**(4) 네트워크 패킷 캡처**
```
네트워크 트래픽 캡처

도구:
- tcpdump
- Wireshark
```

**주의:**
- **원본 보존**
- 모든 작업은 **복사본**에서

### 3.4 상세 분석

#### (1) 침입 경로 분석

**질문:**
- 언제 침입했나? (시간)
- 어디로 침입했나? (경로)
- 무엇을 했나? (행위)
- 어떤 데이터에 접근했나?

**로그 분석:**
```
[웹 로그 분석 예시]

123.456.789.000 - - [01/Apr/2024:14:23:45] 
"POST /login.php HTTP/1.1" 200

→ 로그인 시도

123.456.789.000 - - [01/Apr/2024:14:24:12]
"GET /admin.php?id=1' OR '1'='1 HTTP/1.1" 200

→ SQL Injection 시도!

123.456.789.000 - - [01/Apr/2024:14:25:33]
"GET /download.php?file=customer_data.csv HTTP/1.1" 200

→ 파일 다운로드!
```

**시간대 분석:**
```
14:23 - 로그인 시도 (실패)
14:24 - SQL Injection 공격
14:25 - 관리자 페이지 접근
14:26 - 고객 데이터 조회
14:27 - 파일 다운로드
14:28 - 로그아웃
```

#### (2) 피해 범위 확인

**확인 사항:**
- [ ] 유출된 데이터 종류
- [ ] 유출된 데이터 건수
- [ ] 유출된 개인정보 항목
- [ ] 민감정보 포함 여부
- [ ] 고유식별정보 포함 여부

**예시:**
```
[피해 범위 조사 결과]

유출 파일: customer_data.csv
유출 건수: 50,000건
유출 항목:
- 이름
- 주민등록번호 (고유식별정보!)
- 전화번호
- 이메일
- 주소

민감정보: 없음
고유식별정보: 있음 (주민등록번호)

결론: 법적 신고 대상
```

#### (3) 공격자 프로파일링

**IP 추적:**
```
공격 IP: 123.456.789.000

→ 역추적
→ 해외 IP (미국)
→ VPN 사용 추정
```

**공격 도구:**
```
User-Agent 분석
→ sqlmap (SQL Injection 도구)
```

**공격 기법:**
```
- SQL Injection
- 파일 다운로드
- 자동화 툴 사용
```

#### (4) 취약점 확인

**어떻게 침입했나?**
```
취약점: SQL Injection

위치: login.php
코드:
$query = "SELECT * FROM users WHERE id='$id' AND pw='$pw'";

→ 입력값 검증 없음
→ SQL Injection 가능
```

## 4. 3단계: 억제 (Containment)

### 4.1 단기 억제 (Short-term)

**목적:**
추가 피해 **즉시 차단**

**조치:**
- [x] 해당 서버 네트워크 격리
- [x] 공격 IP 차단 (방화벽)
- [x] 취약한 서비스 중단
- [x] 계정 비활성화

### 4.2 장기 억제 (Long-term)

**목적:**
정상 서비스 유지하면서 추가 피해 차단

**조치:**
- [ ] 취약점 임시 패치
- [ ] 접근통제 강화
- [ ] 모니터링 강화

**예시:**
```
SQL Injection 취약점 임시 패치:

$id = mysqli_real_escape_string($conn, $_POST['id']);
$pw = mysqli_real_escape_string($conn, $_POST['pw']);

→ 입력값 검증 추가
```

## 5. 4단계: 제거 (Eradication)

### 5.1 악성코드 제거

**전체 시스템 검사:**
```
백신 전체 스캔
악성코드 탐지 → 삭제
```

### 5.2 취약점 제거

**영구 패치:**
```
SQL Injection 취약점 완전 제거

→ Prepared Statement 사용
$stmt = $conn->prepare("SELECT * FROM users WHERE id=? AND pw=?");
$stmt->bind_param("ss", $id, $pw);
```

### 5.3 백도어 제거

**확인:**
- 불필요한 계정 삭제
- 의심 파일 삭제
- 스케줄러 확인

## 6. 5단계: 복구 (Recovery)

### 6.1 시스템 복구

#### 복구 방법 결정

**옵션 1: 백업 복구**
```
장점:
- 빠름
- 확실함

단점:
- 최신 데이터 손실 가능

조건:
- 깨끗한 백업 있음
- 백업 시점 이후 데이터 손실 감수 가능
```

**옵션 2: 재설치**
```
장점:
- 완전히 깨끗함

단점:
- 시간 소요
- 데이터 손실

조건:
- 백업 없음 or 백업도 감염
- 완전 복구 필요
```

**옵션 3: 수동 복구**
```
장점:
- 데이터 보존

단점:
- 시간 소요
- 불완전할 수 있음

조건:
- 백업 없고 데이터 중요
```

#### 복구 절차
```
1. 시스템 점검
   - 백도어 제거 확인
   - 취약점 패치 확인
   ↓
2. 백업 복구 (또는 재설치)
   ↓
3. 패치 적용
   - 최신 보안 패치
   - 취약점 패치
   ↓
4. 보안 설정 강화
   - 방화벽 정책
   - 접근통제
   ↓
5. 테스트
   - 기능 테스트
   - 보안 테스트
   ↓
6. 모니터링
   - 재침입 감시
```

### 6.2 서비스 재개

**단계적 재개:**
```
1. 내부 테스트 (1~2일)
   ↓
2. 베타 오픈 (일부 사용자, 1~2일)
   ↓
3. 전체 오픈
```

**모니터링 강화:**
- 재침입 시도 감시
- 이상 징후 즉시 보고

## 7. 6단계: 사후 조치

### 7.1 법적 신고

#### 개인정보 유출 시

**신고 대상:**

**(1) 개인정보보호위원회 (또는 KISA)**

**신고 시기:**
**유출 사실 확인 즉시** (24시간 이내 권장)

**신고 방법:**
- 개인정보보호 종합포털: https://www.privacy.go.kr
- KISA: 국번없이 118

**신고 내용:**
```
[개인정보 유출 신고]

1. 기업 정보
   - 회사명: ㈜ABC
   - 대표자: 홍길동
   - 사업자번호: 123-45-67890

2. 유출 개요
   - 발생 일시: 2024.4.1 14:00~15:00
   - 탐지 일시: 2024.4.1 15:30
   - 유출 경로: 웹서버 해킹 (SQL Injection)

3. 유출 규모
   - 건수: 50,000건
   - 항목: 이름, 주민등록번호, 전화번호, 이메일, 주소

4. 조치 사항
   - 시스템 격리
   - 취약점 패치
   - 정보주체 통지 (진행 중)
```

**(2) 경찰**

**고소 여부 결정:**
- 가해자 처벌 원하면 고소
- 증거 제출 (로그, 포렌식 결과)

#### 신고 기한

**개인정보보호법 제34조:**
"개인정보 유출 사실을 안 때부터 **지체 없이** 해당 정보주체에게 알려야 한다"

**실무:**
- 유출 확인 후 **24시간 이내** 신고 권장
- 늦어도 **3일 이내**

### 7.2 정보주체 통지

#### 통지 의무

**개인정보보호법 제34조:**
개인정보 유출 시 정보주체에게 **통지** 의무

#### 통지 내용

**필수 항목:**
1. 유출된 개인정보 항목
2. 유출 시점 및 경위
3. 유출로 인해 발생할 수 있는 피해 최소화 방법
4. 대응 조치 및 피해 구제 절차
5. 정보주체에게 피해가 발생한 경우 신고 등을 접수할 수 있는 담당 부서 및 연락처

#### 통지 방법

**개별 통지:**
- 이메일
- SMS
- 우편
- 전화

**예시:**
```
[개인정보 유출 안내]

㈜ABC를 이용해 주셔서 감사합니다.

당사는 2024년 4월 1일 외부 해킹으로 인해
고객님의 개인정보가 유출되었음을 알려드립니다.

1. 유출된 개인정보 항목
   - 이름, 주민등록번호, 전화번호, 이메일, 주소

2. 유출 시점 및 경위
   - 발생: 2024년 4월 1일 14:00~15:00
   - 경위: 외부 해커가 웹사이트 취약점을 이용하여
           데이터베이스에 접근

3. 피해 최소화 방법
   - 비밀번호를 즉시 변경해 주시기 바랍니다
   - 의심스러운 연락(보이스피싱 등)에 주의하시기 바랍니다
   - 금융거래 내역을 주기적으로 확인하시기 바랍니다

4. 당사 대응 조치
   - 시스템 취약점 패치 완료
   - 보안 강화
   - 경찰에 고소

5. 피해 구제 절차
   - 피해가 발생한 경우 당사 고객센터로 연락 주시기 바랍니다
   - 전화: 1588-0000
   - 이메일: help@abc.com

대단히 죄송합니다.
㈜ABC 드림
```

**목차:**

## 9. 침해사고 유형별 대응

### 9.1 랜섬웨어

#### 특징
- 파일 암호화
- 금전 요구

#### 대응

**(1) 즉시 조치**
```
- 감염 PC 네트워크 격리 (즉시!)
- 전원 끄지 말 것 (복구 가능성)
- 백업 확인
```

**(2) 복구**
```
옵션 1: 백업 복구 (권장)
옵션 2: 복호화 툴 시도 (No More Ransom 등)
옵션 3: 돈 지불 (비권장!)
```

**(3) 신고**
```
- 경찰 고소
- KISA 신고
```

### 9.2 DDoS 공격

#### 특징
- 서비스 마비
- 트래픽 폭증

#### 대응

**(1) 즉시 조치**
```
- ISP에 연락 (트래픽 차단)
- CDN 활성화 (Cloudflare 등)
- 불필요한 서비스 중단
```

**(2) 완화**
```
- Rate Limiting
- IP 차단
```

**(3) 복구**
```
- 공격 종료 확인
- 서비스 재개
```

### 9.3 내부자 유출

#### 특징
- 직원이 고의로 정보 유출
- 발견 어려움

#### 대응

**(1) 즉시 조치**
```
- 해당 직원 계정 즉시 비활성화
- 접근 권한 박탈
- 출입 제한
```

**(2) 조사**
```
- 로그 분석 (언제, 무엇을, 어떻게)
- 유출 범위 확인
```

**(3) 법적 조치**
```
- 경찰 고소
- 징계 (해고)
- 손해배상 청구
```

### 9.4 피싱 (Phishing)

#### 특징
- 가짜 사이트로 유인
- 개인정보 입력 유도

#### 대응

**(1) 즉시 조치**
```
- 가짜 사이트 URL 파악
- ISP에 차단 요청
```

**(2) 고객 안내**
```
- 피싱 사이트 경고
- 비밀번호 변경 권고
```

**(3) 신고**
```
- 경찰 고소
- KISA 신고
```

## 학습 정리

오늘 학습한 핵심 내용:
- 침해사고 대응 6단계: 준비 → 탐지 → 억제 → 제거 → 복구 → 사후
- 사전 준비: CSIRT 구성, 대응 절차 수립, 도구 준비, 훈련
- 탐지: 자동 탐지, 모니터링, 신고
- 초동 대응: 격리, 계정 차단, 증거 보존 (로그, 메모리, 디스크)
- 분석: 침입 경로, 피해 범위, 취약점 확인
- 억제: 추가 피해 차단
- 제거: 악성코드 제거, 취약점 패치, 백도어 제거
- 복구: 백업 복구 또는 재설치, 단계적 재개
- 사후: 법적 신고(개보위, 경찰), 정보주체 통지, 사고 보고서, 재발 방지
- 개인정보 유출 시 즉시 신고 의무 (24시간 이내 권장)
- 정보주체 통지 의무 (개별 통지)
- 유형별 대응: 랜섬웨어(격리, 백업 복구), DDoS(ISP 연락), 내부자(계정 차단, 고소)
