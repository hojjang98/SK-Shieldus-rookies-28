# Day 44: 실무 사례 #4 - SK컴즈 네이트·싸이월드 해킹 (2011)

## 1. 사건 개요

### 1.1 기본 정보

회사: SK컴즈 (현 SK플래닛)
서비스: 네이트 (포털), 싸이월드 (SNS)
발생 시기: 2011년 7월 26일
발견: 2011년 7월 26일
유출 규모: 약 3,500만 건 (당시 국내 최대)

사건 요약:
중국인 해커가 SK컴즈 네이트·싸이월드 서버를 해킹하여 회원 개인정보 3,500만 건을 유출했다. 유출 정보에는 이름, 주민등록번호, 이메일, 전화번호 등이 포함되었으며, 당시 대한민국 인구의 70%에 해당하는 사상 최대 규모였다.

### 1.2 상세 경위

공격 시점: 2011년 7월 25일 ~ 7월 26일

공격 방법:
1. 웹 서버 취약점 공격 - SQL Injection 공격, 웹 애플리케이션 보안 취약점 악용
2. 데이터베이스 접근 - 관리자 권한 탈취, 회원 DB 서버 접근
3. 대량 다운로드 - 3,500만 건 회원 정보 다운로드, 약 7.8GB 데이터

유출 데이터:
이름 3,500만 건, 주민등록번호 3,500만 건, 전화번호 3,500만 건, 이메일 3,500만 건, 주소 일부, 비밀번호 암호화되어 있었으나 일부 해독 가능성

유출 규모 의미:
2011년 대한민국 인구 약 5,000만명, 유출 3,500만 건으로 인구의 70% 해당. 실제 피해자는 중복 가입 고려 시 약 2,000만~2,500만명 추정으로 대한민국 인구의 40~50%에 해당.

## 2. 법률 분석

### 2.1 적용 법률

주요 법률:
1. 개인정보보호법 (2011년 9월 30일 시행 직전)
2. 정보통신망법
3. 형법

특이 사항:

### 2.2 위반 사항 분석

(1) 안전성 확보 조치 의무 위반

법적 근거: 정보통신망법 제28조 (개인정보의 기술적·관리적 보호조치)

당시 규정: 정보통신서비스 제공자는 이용자의 개인정보를 취급함에 있어서 개인정보의 분실·도난·유출·변조 또는 훼손을 방지하기 위하여 대통령령이 정하는 기준에 따라 기술적·관리적 조치를 하여야 한다.

SK컴즈 위반 내용:
1. 웹 애플리케이션 보안 취약 - SQL Injection 취약점 존재, 입력값 검증 미흡, Prepared Statement 미사용
2. 주민등록번호 암호화 미흡 - 평문 저장 또는 취약한 암호화
3. 침입탐지 지연 - 공격 시작부터 탐지까지 약 2~3시간 지연, 이미 다운로드 완료 후 탐지
4. 접근통제 미흡 - 관리자 권한 탈취 허용, 대량 데이터 다운로드 차단 실패
5. 로그 모니터링 부족 - 비정상 접근 패턴 사전 탐지 실패

해당 조항: 정보통신망법 제73조 (벌칙) - 제28조를 위반하여 기술적·관리적 조치를 하지 아니하여 이용자의 개인정보를 분실·도난·유출·변조 또는 훼손당한 자는 2년 이하의 징역 또는 2천만원 이하의 벌금에 처한다.

(2) 개인정보 유출 통지 의무

법적 근거: 정보통신망법 제27조의3 (개인정보 유출 등의 통지)

당시 규정: 정보통신서비스 제공자는 개인정보의 유출 사실을 안 때에는 지체 없이 해당 이용자에게 통지해야 한다.

SK컴즈 대응: 유출 7월 26일 새벽, 통지 7월 27일로 약 1일 소요. 당시 기준으로는 비교적 신속했으나 현재 기준으로는 지연 (24시간 내 권장).

(3) 해커에 대한 처벌

법적 근거:
- 정보통신망법 제48조 (정보통신망 침해)
- 개인정보보호법 제70조 (개인정보 부정 이용, 시행 전이지만 적용 가능)
- 형법 제314조의2 (컴퓨터등장애업무방해)

중국인 해커: 이름 미공개 중국인, 20대. 혐의는 정보통신망 침해 5년 이하, 개인정보 부정 이용 5년 이하, 컴퓨터등장애업무방해 5년 이하. 중국 공안이 검거했으나 한국 송환 안 됨 (범죄인 인도 협정 부재), 중국 법으로 처벌 (형량 미공개).

### 2.3 처벌 및 제재

SK컴즈 제재

방송통신위원회 (당시 주무 부처):
과징금 미부과 (당시 정보통신망법에 과징금 규정 없음)
시정명령: 보안 강화, 재발 방지 대책 수립, 정기 보고
경고: 중대한 보안 사고, 관리 책임 추궁

형사 처벌: 안전조치 의무 위반으로 2년 이하 징역 또는 2천만원 이하 벌금. 실제 처벌 미공개 (법인 및 담당자에 대한 형사 처벌 여부 불명).

민사 배상:
집단 소송 (2011년 9월 제기) - 원고 약 100만명, 청구액 1인당 100만원으로 총 1조원 규모
1심 판결 (2014년) - 1인당 20만원 배상 (일부 인용), 총 배상액 약 200억원
2심 판결 (2016년) - 1인당 10만원~20만원 배상 (감액), 최종 배상액 약 100억원~150억원 추정

총 비용:
민사 배상 약 100~150억원, 시스템 개선 약 200억원 (추정), 평판 손실 측정 불가, 서비스 이탈 약 500만명 (추정)
총 직접 비용 약 300~350억원, 간접 비용 수천억원 (추정)

사업 영향

싸이월드 몰락:
유출 사건 전 - 가입자 약 2,500만명, 국내 1위 SNS
유출 사건 후 - 고객 이탈 급증, 페이스북·카카오스토리로 이동, 2015년 서비스 사실상 중단, 2019년 일시 재개했으나 실패
원인 분석 - 유출 사건이 결정타, 신뢰 회복 실패, 경쟁 서비스 부상

네이트 쇠퇴:
유출 사건 전 - 국내 3위 포털 (네이버, 다음 다음)
유출 사건 후 - 점유율 급락. 2011년 약 15%, 2015년 약 5%, 2020년 약 1% 미만

## 3. 기술적 분석

### 3.1 공격 기법: SQL Injection

공격 과정:
1. 취약점 발견 - 네이트 또는 싸이월드 로그인 페이지 또는 검색 기능에서 SQL Injection 취약점 존재
2. SQL 쿼리 조작 - 정상 로그인 SELECT * FROM users WHERE id='hong' AND pw='1234'를 해킹 로그인 id: admin' OR '1'='1' -- 로 조작하여 SELECT * FROM users WHERE id='admin' OR '1'='1' -- AND pw='...' 실행, 로그인 성공
3. 데이터베이스 탐색 - UNION SELECT 사용, 테이블 구조 파악, 회원 테이블 발견
4. 대량 다운로드 - SELECT * FROM members로 3,500만 건 조회, 파일로 저장 (7.8GB)

악용 예시:
회원 정보 조회: ' UNION SELECT name, ssn, phone, email FROM members --
관리자 권한 획득: ' OR '1'='1' --
데이터베이스 정보 추출: ' UNION SELECT database(), user(), version() --

### 3.2 SK컴즈의 보안 취약점

(1) 웹 애플리케이션 보안 미흡

문제: SQL Injection 취약점 존재, 가장 기본적인 웹 취약점
원인: 입력값 검증 없음, Prepared Statement 미사용, 웹 방화벽 (WAF) 없음

올바른 코드 (PHP 예시):
취약한 코드:
$id = $_POST['id'];
$pw = $_POST['pw'];
$query = "SELECT * FROM users WHERE id='$id' AND pw='$pw'";

안전한 코드:
$stmt = $conn->prepare("SELECT * FROM users WHERE id=? AND pw=?");
$stmt->bind_param("ss", $id, $pw);
$stmt->execute();

(2) 주민등록번호 암호화 미흡

문제: 2011년 당시 SK컴즈는 주민등록번호를 평문 또는 단순 암호화로 저장
이유: 2000년대 초반 시스템 구축 당시 암호화 의무 없었음, 개인정보보호법 시행 전 (2011년 9월), 레거시 시스템 개선 지연

당시 법적 상황:
2011년 7월 (유출 당시) - 개인정보보호법 미시행, 정보통신망법에 암호화 의무 있었으나 구체적 기준 모호
2011년 9월 (개인정보보호법 시행) - 주민등록번호 암호화 의무 명확화, 안전성 확보조치 기준 구체화

(3) 침입탐지 지연

문제: 공격 시작 7월 25일 밤, 탐지 7월 26일 새벽 03:30으로 수시간 지연, 이미 3,500만 건 다운로드 완료 후 탐지

원인: 실시간 모니터링 부재, 비정상 트래픽 탐지 시스템 미흡, 대량 다운로드 경고 설정 없음

필요한 대책:
1. 실시간 IDS/IPS
2. 대량 조회 시 자동 경고 (예: 1시간 내 10만 건 이상 조회)
3. 비정상 접근 패턴 탐지 (예: 새벽 시간대 관리자 로그인)
4. SIEM (보안 정보 통합 관리)

(4) 접근통제 미흡

문제: 해커가 관리자 권한으로 전체 회원 DB 무제한 접근 가능

필요한 통제:
1. 최소 권한 원칙 - 관리자도 필요한 정보만
2. 접근 승인 - 대량 데이터 조회 시 상급자 승인
3. 조회 건수 제한 - 일일 조회 건수 상한
4. IP 화이트리스트 - 특정 IP에서만 관리자 접근 허용

### 3.3 사후 SK컴즈 보안 강화

2011년 이후 조치:

1. 웹 보안 강화
- 전사 웹 취약점 진단
- SQL Injection 취약점 전면 제거
- Prepared Statement 전환
- 웹 방화벽 (WAF) 도입
- 정기 모의해킹 (분기 1회)

2. 암호화 강화
- 주민등록번호 AES-256 암호화
- 비밀번호 SHA-256 해시
- 전송 구간 TLS 1.2

3. 침입탐지 강화
- IDS/IPS 전면 도입
- SIEM 구축
- 24/365 보안관제센터
- 실시간 위협 탐지

4. 접근통제 강화
- 최소 권한 원칙
- 대량 조회 시 경고
- IP 화이트리스트
- 이중 인증 (2FA)

5. 조직 강화
- CISO 신설 (임원급)
- 보안팀 인력 증원
- 보안 예산 대폭 확대

## 4. 대응 평가

### 4.1 잘한 점

신속한 신고: 유출 확인 후 수시간 내 방송통신위원회 신고, 개인정보보호위원회 신고, 경찰 신고

투명한 공개: 언론 브리핑, 홈페이지 공지, 고객 통지 (이메일, SMS), 대표 공식 사과

해커 검거 협력: 중국 공안과 협력, 해커 검거 성공

### 4.2 미흡한 점

사전 예방 실패:
1. SQL Injection 취약점 방치 - 가장 기본적인 웹 취약점, OWASP Top 10 1위
2. 주민등록번호 평문 저장 - 개인정보보호법 시행 직전까지 방치, 레거시 개선 지연
3. 침입탐지 지연 - 실시간 모니터링 부재, 3,500만 건 다운로드 후 탐지

규모: 3,500만 건은 당시 국내 최대, 사회적 충격 심각, 개인정보 보호 인식 전환점

신뢰 회복 실패: 싸이월드 사실상 폐업, 네이트 점유율 급락, SK컴즈 브랜드 이미지 추락

## 5. 교훈 및 시사점

### 5.1 기술적 교훈

1. SQL Injection은 여전히 위협

2011년에도, 2024년에도 여전히 발생. 이유는 개발자 보안 인식 부족, 레거시 코드, 빠른 개발 압박.
대응: 입력값 검증 필수, Prepared Statement 필수, 웹 방화벽 (WAF), 정기 취약점 진단, 개발자 시큐어 코딩 교육

2. 주민등록번호는 특급 관리

유출 시 평생 사용 불가, 재발급 불가능.
대응: AES-256 암호화 필수, 가능하면 수집 안 함, 대체 수단 (아이핀, 휴대폰 인증)

3. 실시간 모니터링

3,500만 건 다운로드는 약 7.8GB로 최소 수십 분 이상 소요, 실시간 탐지 가능했음.
대응: 대량 조회 시 자동 경고, 비정상 패턴 탐지, SIEM

4. 레거시 시스템 개선

오래된 시스템은 취약점 온상.
대응: 정기적 보안 점검, 단계적 개선, 리팩토링

### 5.2 법률적 교훈

1. 개인정보보호법 시행의 계기

SK컴즈 사건 발생 2011년 7월, 개인정보보호법 시행 2011년 9월 (2개월 후). SK컴즈 사건이 개인정보 보호 강화의 계기가 되어 개인정보보호법 시행을 앞당기고 사회적 인식을 전환시켰다.

2. 과징금 제도 도입

당시는 과징금 없음 (시정명령만), 이후 매출액 3% 이하 과징금 도입. SK컴즈가 2015년 발생했다면 과징금 수백억원 (매출 고려).

3. 암호화 의무 강화

이전에는 모호한 기준, 이후 명확한 기준 - AES-256, ARIA-256 등, SHA-256 이상 해시, TLS 1.2 이상.

4. 집단 소송 활성화

SK컴즈 사건은 국내 최대 규모 집단 소송. 원고 100만명, 청구액 1조원. 이후 개인정보 유출 시 집단 소송 일반화, 손해배상 인정 (1인당 10만원~20만원).

### 5.3 경영적 교훈

1. 보안 사고는 기업 생존 위협

싸이월드 - 유출 전 국내 1위 SNS 가입자 2,500만명, 유출 후 서비스 중단 (2015년). 원인은 유출 사건이 결정타, 신뢰 회복 실패, 고객 대량 이탈.

2. 평판 손실은 측정 불가

직접 비용 300~350억원, 간접 비용은 고객 이탈, 매출 감소, 브랜드 가치 하락으로 수천억원 추정. SK컴즈는 사건 후 사업 재편, 싸이월드 매각 시도 (실패), SK플래닛으로 합병.

3. 사전 투자의 중요성

사후 비용 300~350억원 (직접) + 수천억원 (간접). 사전 투자 (추정) - 웹 방화벽 5천만원, 암호화 개선 10억원, 취약점 진단 1억원으로 총 12억원. 300억원 vs 12억원은 25배 차이.

### 5.4 사회적 영향

개인정보 보호 인식 전환:
SK컴즈 사건 이전 - 주민번호 무분별 수집, 개인정보 보호 인식 낮음
SK컴즈 사건 이후 - 주민번호 수집 자제, 개인정보 보호 중요성 인식, 대체 인증 수단 확산 (아이핀, 휴대폰)

법제도 강화:
개인정보보호법 시행 가속화, 과징금 제도 도입, 암호화 기준 명확화, 집단 소송 활성화

## 9. 실무 적용

### 9.1 SQL Injection 방어 체크리스트

코드 레벨 (PHP):
Bad: $query = "SELECT * FROM users WHERE id='$id'";
Good: $stmt = $conn->prepare("SELECT * FROM users WHERE id=?"); $stmt->bind_param("s", $id);

입력값 검증:
화이트리스트 방식: $id = preg_replace('/[^a-zA-Z0-9_]/', '', $_POST['id']);
길이 제한: if (strlen($id) > 20) { die("Invalid input"); }

웹 방화벽 (WAF): SQL Injection 패턴 차단 - ' OR '1'='1, UNION SELECT, DROP TABLE 등

### 9.2 대량 조회 탐지 정책

경고 기준:
Level 1 (경고) - 1시간 내 1,000건 이상 조회, 보안팀 알림
Level 2 (차단) - 1시간 내 10,000건 이상 조회, 자동 계정 잠금
Level 3 (긴급) - 1시간 내 100,000건 이상 조회, CSIRT 소집

모니터링 (SQL):
SELECT user_id, COUNT(*) as cnt
FROM access_log
WHERE created_at > NOW() - INTERVAL 1 HOUR
GROUP BY user_id
HAVING cnt > 1000
ORDER BY cnt DESC;

### 9.3 개발자 시큐어 코딩 교육

필수 교육 내용:

1. OWASP Top 10 - SQL Injection, XSS (Cross-Site Scripting), CSRF (Cross-Site Request Forgery) 등

2. 입력값 검증 - 화이트리스트, 길이 제한, 타입 검사

3. Prepared Statement - 모든 SQL 쿼리는 Prepared Statement, 예외 없음

4. 암호화 - 비밀번호는 해시 + Salt, 주민번호는 AES-256, 전송은 TLS 1.2

5. 실습 - 취약한 코드 찾기, 안전한 코드 작성

## 학습 정리

SK컴즈 사건에서 배운 점:
- SQL Injection은 가장 기본적이지만 가장 위험한 웹 취약점
- 3,500만 건 유출은 국내 역사상 최대 규모 (인구의 70%)
- 주민등록번호 평문 저장은 치명적 (암호화 필수)
- 실시간 모니터링으로 대량 다운로드 탐지 가능했음
- 개인정보보호법 시행 직전 사건 (2개월 차이)으로 과징금 없었음
- 집단 소송 100만명 참여, 배상 약 100~150억원
- 싸이월드 폐업, 네이트 몰락으로 기업 생존 위협
- 개인정보 보호 인식 전환의 계기
- Prepared Statement, WAF, 실시간 모니터링 필수
- 사전 투자 12억원으로 300억원 이상 손실 예방 가능 (25배 차이)
- 레거시 시스템 보안 개선 미룰수록 위험 증가
