# Day 48: 실무 사례 #8 - 웹젠·나야나 랜섬웨어 공격 (2017)

## 1. 사건 개요

### 1.1 기본 정보

피해 기업: 웹젠 (게임사), 나야나 (호스팅사)
발생 시기: 2017년 6월
사건 유형: 랜섬웨어 공격 (암호화 + 금전 요구)
랜섬웨어 종류: Erebus (에레버스)
피해 규모: 서버 153대 암호화, 웹사이트 3,400개 마비
몸값 지불: 약 13억원 (550 비트코인)

사건 요약:
2017년 6월, 웹 호스팅 업체 나야나가 랜섬웨어 공격을 받아 서버 153대가 암호화되었다. 나야나를 이용하는 웹사이트 약 3,400개가 동시에 마비되었고, 이 중에는 게임사 웹젠의 공식 홈페이지도 포함되어 있었다. 나야나는 결국 해커에게 약 13억원 (550 비트코인)의 몸값을 지불하고 복호화 키를 받았다. 국내 최초로 대규모 랜섬웨어 공격에 몸값을 지불한 사례로, 랜섬웨어 위협의 심각성을 알린 사건이다.

### 1.2 상세 경위

공격 시점: 2017년 6월 10일

공격 대상:
주 피해자: 나야나 (인터넷나야나, 웹 호스팅 업체)
간접 피해자: 나야나 고객사 약 3,400개 (웹젠 포함)

공격 방법:
1. 웹 서버 취약점 공격: Apache Struts 취약점 악용 (추정), 또는 관리자 계정 탈취
2. 랜섬웨어 설치: Erebus 랜섬웨어 설치, 서버 153대에 확산
3. 파일 암호화: 모든 데이터 파일 암호화 (웹사이트, 데이터베이스, 백업 파일 포함), 암호화 알고리즘 RSA-2048 + AES-128
4. 몸값 요구: 화면에 랜섬 노트 표시, "복호화 원하면 비트코인 지불하라", 초기 요구액 550 비트코인 (당시 약 15억원)

피해 상황:
나야나 서버 153대 암호화. 고객사 웹사이트 3,400개 접속 불가. 웹젠 공식 홈페이지 마비. 중소 쇼핑몰, 기업 홈페이지 등 동시 마비. 데이터베이스 암호화로 고객 정보, 거래 내역 등 접근 불가.

협상 과정:
나야나는 해커와 협상 시도. 초기 요구 550 BTC → 최종 합의 397.6 BTC (약 13억원). 3차례 분할 지불 (1차 100 BTC, 2차 100 BTC, 3차 197.6 BTC).

복구 과정:
1차 지불 후 일부 복호화 키 제공. 2차 지불 후 추가 복호화 키. 3차 지불 후 최종 복호화 키. 복호화 작업 약 2주 소요. 일부 데이터는 복구 실패 (손상).

대부분 서비스 복구 완료. 일부 고객 데이터 손실 (약 5%).

2017년 7월 ~ 8월
경찰 수사 진행. 해커 추적 (해외, 검거 실패).

2호: 정보통신망에 장애가 발생하게 하거나 정보를 훼손하는 등 악성프로그램을 전달 또는 유포한 자

처벌: 5년 이하 징역

해당 사항: 데이터 암호화 = 전자기록 효용 침해.

## 3. 기술적 분석

### 3.1 랜섬웨어란?

정의: 사용자 파일을 암호화하고 복호화 대가로 몸값 (Ransom)을 요구하는 악성프로그램.

동작 방식:
1. 감염: 이메일 첨부파일, 웹사이트 취약점, 관리자 계정 탈취 등
2. 확산: 네트워크 공유 폴더, 연결된 드라이브로 확산
3. 암호화: 문서, 이미지, 데이터베이스 등 중요 파일 암호화
4. 몸값 요구: 화면에 랜섬 노트 표시, 비트코인 지불 요구
5. 복호화 (지불 시): 몸값 지불 시 복호화 키 제공 (때로는 거짓)

암호화 알고리즘:
대부분 RSA-2048 이상 + AES-128/256. 복호화 불가능 (현재 기술로는 수억 년 소요).

주요 랜섬웨어:
WannaCry (2017, 전 세계), Petya/NotPetya (2017), Erebus (나야나, 2017), GandCrab (2018~2019), REvil/Sodinokibi (2019~2021).

### 3.2 Erebus 랜섬웨어 분석

개발: 2016년 발견, 초기에는 Windows 공격, 2017년 Linux 버전 등장.

나야나 공격 버전: Linux Erebus (서버 타겟).

감염 경로 (추정):
Apache Struts 취약점 (CVE-2017-5638): 원격 코드 실행 취약점, 2017년 3월 발견, 패치 공개, 나야나는 패치 지연 (추정).
또는 관리자 계정 탈취: 약한 비밀번호, SSH 무차별 대입 공격.

암호화 방식:
RSA-2048 + AES-128. 파일마다 고유 AES 키 생성. RSA 공개키로 AES 키 암호화. 복호화는 RSA 개인키 필요 (해커만 보유).

암호화 대상:
웹사이트 파일 (HTML, PHP, JS 등), 데이터베이스 파일 (MySQL, PostgreSQL), 이미지, 백업 파일, 로그 파일.

암호화 제외:
시스템 파일 (부팅 가능하도록), 실행 파일 일부 (랜섬웨어 자체 동작 위해).

랜섬 노트:
화면에 표시: "Your files are encrypted", "Send 550 BTC to [비트코인 주소]", "After payment, contact [이메일]".

특징:
빠른 암호화 속도 (153대 서버를 수시간 내). 백업 파일도 암호화 (동일 서버 저장 시). 네트워크 확산 기능 (SSH, SMB 등).

### 3.3 나야나의 보안 취약점

(1) 서버 보안 미흡

문제:
Apache Struts 취약점 패치 지연 (추정). CVE-2017-5638은 2017년 3월 공개. 나야나 공격은 6월. 3개월간 패치 안 함.

원인:
패치 관리 부실. 취약점 정보 모니터링 부족. 보안 인력 부족.

필요한 대책:
취약점 정보 모니터링 (KISA, 보안 업체). 긴급 패치 즉시 적용. 정기 취약점 스캔.

(2) 백업 실패

문제:
백업 파일도 동일 서버에 저장. 랜섬웨어가 백업도 암호화. 복구 불가능.

원인:
오프라인 백업 없음. 원격지 백업 없음. 백업 전략 부재.

필요한 대책:
3-2-1 백업 원칙 (3개 복사본, 2개 매체, 1개 원격지). 오프라인 백업 (에어갭). 백업 테스트 정기 실시.

(3) 랜섬웨어 대응 미흡

문제:
백신 미설치 또는 업데이트 지연. 랜섬웨어 탐지 실패. 확산 차단 실패.

원인:
서버용 백신 미구축. 네트워크 세그먼트 부족. 실시간 모니터링 없음.

필요한 대책:
서버용 백신 설치. EDR (Endpoint Detection and Response). 네트워크 세그먼트. 실시간 모니터링.

(4) 관리자 계정 보안

문제:
관리자 계정 탈취 가능성. 약한 비밀번호 또는 이중 인증 없음.

필요한 대책:
강력한 비밀번호 (16자 이상). 이중 인증 (2FA). SSH 키 인증. IP 화이트리스트.

### 3.4 사후 대응

복호화 과정:
해커로부터 복호화 키 수령. 서버별 순차 복호화. 복호화 도구 실행 (해커 제공). 파일 무결성 검증. 일부 파일 손상 확인 (약 5%).

복호화 소요 시간:
서버 153대 복호화 약 2주. 서버당 평균 1~2일.

데이터 손실:
약 5% 데이터 복구 실패. 원인: 복호화 과정 오류, 파일 손상, 복호화 키 불완전.

보안 강화 (이후):
서버 전면 재구축. 최신 패치 적용. 백신 설치. 오프라인 백업 체계 구축. 보안 인력 충원. 정기 취약점 스캔.

## 4. 대응 평가

### 4.1 잘한 점

신속한 신고:
발견 즉시 경찰, KISA 신고. 법적 의무 준수.

투명한 공개:
고객에게 즉시 공지. 랜섬웨어 공격 사실 공개. 복구 진행 상황 실시간 공유.

고객 보상:
데이터 손실 고객에게 일부 보상 (금액 미공개). 서비스 이용료 환불.

### 4.2 미흡한 점

사전 예방 실패:
1. 취약점 패치 지연 (3개월)
2. 백업 체계 부재 (오프라인 백업 없음)
3. 랜섬웨어 대응 미흡

몸값 지불:
13억원 몸값 지불. 해커에게 자금 제공. 향후 공격 유인 (랜섬웨어는 돈이 된다는 인식).

복구 지연:
2주 소요. 고객 피해 장기화.

고객 신뢰 상실:
사건 후 고객 30% 이탈. 회사 사실상 폐업.

## 5. 교훈 및 시사점

### 5.1 기술적 교훈

1. 랜섬웨어는 복호화 불가능

RSA-2048 이상 암호화 = 현재 기술로 복호화 불가능. 몸값 지불 외 방법 없음 (백업 있으면 제외).

대응: 백업이 유일한 해법. 오프라인 백업 필수.

2. 백업은 생명줄

백업 없으면 복구 불가능. 몸값 지불하거나 데이터 포기.

대응: 3-2-1 원칙. 오프라인 백업 (에어갭). 백업 테스트.

3. 취약점 패치는 즉시

Apache Struts 취약점 3개월 방치 = 사고 초대.

대응: 취약점 발견 즉시 패치. 긴급 패치는 24시간 내. 정기 스캔.

4. 랜섬웨어는 확산 빠름

네트워크 공유, SSH, SMB 등으로 빠르게 확산. 수시간 내 전체 암호화.

대응: 네트워크 세그먼트. 감염 PC 즉시 격리. 네트워크 차단.

5. 서버도 백신 필수

서버는 백신 없는 경우 많음 (성능 저하 우려). 하지만 랜섬웨어는 서버도 공격.

대응: 서버용 백신 설치. EDR. 실시간 모니터링.

### 5.2 법률적 교훈

1. 랜섬웨어는 공갈죄

몸값 요구 = 공갈죄. 10년 이하 징역.

하지만: 해커 대부분 해외, 익명성. 검거 어려움.

2. 몸값 지불의 법적 문제

몸값 지불 = 범죄자에게 자금 제공. 법적으로 문제 소지 (범죄 방조 가능성). 하지만 현실적으로 처벌 사례 없음 (피해자이므로).

논란: 몸값 지불 금지해야 하는가? 일부 국가는 금지 검토. 한국은 아직 금지 안 함.

3. 기업의 안전조치 의무

정보통신망법 제28조. 안전조치 미흡 = 2년 이하 징역.

나야나: 취약점 패치 지연, 백업 부실 = 안전조치 의무 위반 가능.

4. 손해배상 책임

고객이 나야나에 손해배상 청구 가능. 안전조치 의무 위반으로 인한 피해.

실제: 일부 고객 청구, 합의 추정.

### 5.3 경영적 교훈

1. 랜섬웨어 = 기업 생존 위협

나야나 총 손실: 약 30억원 (직접) + 수백억원 (간접). 고객 30% 이탈. 2020년 사실상 폐업.

원인: 신뢰 상실. 고객 이탈. 회복 불가능.

2. 몸값 vs 백업 비용

몸값: 13억원
백업 시스템 구축 비용 (추정): 1억원 (오프라인 백업, 원격지 백업)

13억원 vs 1억원 = 13배 차이

교훈: 백업에 투자하는 것이 훨씬 저렴.

3. 몸값 지불의 딜레마

장점: 데이터 복구 가능 (70~90% 성공률). 빠른 복구 (2주).
단점: 비용 막대 (13억원). 해커 자금 지원 (향후 공격 유인). 복호화 실패 가능성 (약 10~30%). 데이터 일부 손상 (5%).

결론: 백업 있으면 절대 지불하지 말 것. 백업 없으면 선택의 여지 없음.

4. 호스팅 사업의 책임

호스팅 업체 = 고객 데이터 보관. 막중한 책임. 보안 사고 = 수천 고객 동시 피해.

교훈: 호스팅 업체는 최고 수준 보안 필요. 백업 필수. 보안 인력 충분히 확보.

### 5.4 사회적 영향

랜섬웨어 인식 제고:
나야나 사건 = 국내 랜섬웨어 경각심 제고. 기업들 백업 강화. 보안 투자 증가.

몸값 지불 논란:
몸값 지불 공개 = 사회적 논란. "범죄자에게 돈 주는 것 맞나?". "하지만 다른 방법 없지 않나?". 정부 가이드라인 필요성 대두.

중소기업 보안 취약성:
나야나 = 중소기업. 보안 투자 부족. 인력 부족. 중소기업 보안 지원 필요성 인식.

- 백업 테스트 (월 1회)

접근통제:

- IP 화이트리스트
- VPN 사용

네트워크 분리:
- 네트워크 세그먼트
- 중요 서버 격리
- 백업 서버 별도 망

교육:

- 랜섬웨어 인식 교육
- 의심 이메일 신고 체계

사고 대응:

감염 발견 시:
1. 감염 PC 즉시 네트워크 차단
2. 전원 끄지 말 것 (포렌식 증거)
3. 백신으로 스캔
4. KISA 신고 (국번 없이 118)
5. 경찰 신고

복구:
1. 백업에서 복구 (최선)
2. 복호화 도구 확인 (KISA, 백신 업체)

## 9. 실무 적용

### 9.1 랜섬웨어 감염 시 대응 절차

Step 1: 즉시 격리
감염 의심 PC 네트워크 차단 (랜선 뽑기, Wi-Fi 끄기). 다른 PC 감염 방지. 전원은 끄지 말 것 (포렌식 증거 보존).

Step 2: 신고
KISA 신고 (국번 없이 118). 경찰 신고 (사이버안전국). 회사 보안팀 보고.

Step 3: 피해 범위 확인
어떤 파일이 암호화되었나? 몇 대 PC가 감염되었나? 백업은 있나?

Step 4: 복구 방법 결정

옵션 1: 백업에서 복구 (최선)
백업 있으면 즉시 복구. 몸값 지불하지 말 것.

옵션 2: 복호화 도구 (무료)
KISA, 백신 업체 사이트에서 복호화 도구 확인. 일부 랜섬웨어는 복호화 도구 공개.

옵션 3: 몸값 지불 (최후의 수단, 권장 안 함)
백업 없고 복호화 도구 없으면 고려. 하지만 복호화 보장 안 됨 (약 10~30% 실패). 경영진 승인 필요.

Step 5: 재발 방지
원인 분석. 취약점 패치. 백업 체계 구축. 보안 강화.

### 9.2 백업 전략 (3-2-1 원칙)

3개 복사본:
1. 운영 데이터 (Production)
2. 백업 1 (온사이트, 빠른 복구)
3. 백업 2 (오프사이트 또는 클라우드)

2개 매체:
1. 디스크 (빠른 복구)
2. 테이프 또는 클라우드 (장기 보관)

1개 오프사이트:
원격지 백업 또는 클라우드. 화재, 지진 등 물리적 재해 대비.

에어갭 백업:
네트워크에서 완전 분리. 랜섬웨어 방어 최후의 보루.

방법:
백업 후 네트워크 케이블 분리. 또는 전용 백업 장비 (오프라인). 또는 클라우드 불변 백업 (Immutable Backup).

### 9.3 랜섬웨어 복호화 도구

무료 복호화 도구 제공 사이트:
KISA (한국인터넷진흥원): 랜섬웨어 대응 센터
No More Ransom (국제 프로젝트): 다양한 랜섬웨어 복호화 도구
백신 업체 (카스퍼스키, 비트디펜더 등): 일부 랜섬웨어 복호화 도구 제공

주의: 모든 랜섬웨어에 복호화 도구 있는 것 아님. Erebus는 복호화 도구 없음 (2017년 당시).

### 9.4 몸값 지불 시 고려사항

지불 전:
경영진 승인. 법률 자문 (변호사). 경찰 신고 (수사 협조). KISA 상담.

협상:
몸값 인하 시도 (나야나: 550 BTC → 397.6 BTC). 분할 지불 (일부 복호화 후 추가 지불). 복호화 테스트 (일부 파일 먼저).

지불 방법:
비트코인 (익명성, 추적 어려움). 거래소에서 구매. 해커 지갑 주소로 송금.

지불 후:
복호화 키 수령 확인. 복호화 테스트. 전체 복호화 진행. 데이터 무결성 검증.

위험:
복호화 보장 안 됨 (10~30% 실패). 데이터 손상 가능 (5~10%). 해커 재요구 가능 (추가 몸값). 범죄자 자금 지원 (윤리적 문제).

권고: 가능한 한 지불하지 말 것. 백업으로 복구.

## 학습 정리

나야나 사건에서 배운 점:
- 국내 최초 대규모 랜섬웨어 공격 몸값 지불 사례
- 호스팅사 나야나 서버 153대 암호화, 고객사 3,400개 동시 마비
- Erebus 랜섬웨어 (Linux 버전)로 모든 데이터 파일 암호화

- 백업 파일도 동일 서버에 저장되어 함께 암호화됨
- 오프라인 백업 없어 복구 불가능
- 몸값 397.6 BTC (약 13억원) 지불 후 복호화
- 복호화 후에도 약 5% 데이터 손실
- 랜섬웨어는 RSA-2048 암호화로 복호화 불가능
- 백업이 유일한 해법 (3-2-1 원칙, 오프라인 백업 필수)
- 취약점 패치 즉시 적용 필요 (긴급 패치 24시간 내)
- 서버도 백신 설치 필수
- 몸값 지불은 최후의 수단 (권장 안 함, 복호화 보장 안 됨)
- 사건 후 고객 30% 이탈, 회사 사실상 폐업
- 백업 비용 1억원 vs 몸값 13억원 (13배 차이)
