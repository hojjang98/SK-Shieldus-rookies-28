# Day 25: SOC 2 (Service Organization Control 2)

## 1. SOC 2의 개요

### 1.1 SOC 2란?

**정식 명칭**: Service Organization Control 2

**SOC 2**: 서비스 제공 조직의 **내부 통제**를 평가하는 **미국 감사 기준**

**발행 기관**: AICPA (American Institute of Certified Public Accountants, 미국공인회계사협회)

**목적**: 
클라우드 서비스, SaaS, 데이터센터 등 **서비스 제공자**의 보안 및 운영 통제 수준을 고객에게 입증

### 1.2 제정 배경

- **클라우드 컴퓨팅** 급증
- 고객 데이터를 제3자(서비스 제공자)가 처리
- 고객이 서비스 제공자의 **내부 통제 수준**을 확인할 방법 필요
- **표준화된 평가 기준** 요구

### 1.3 SOC 보고서 종류

AICPA는 **3가지 SOC 보고서** 제공:

#### SOC 1
**재무 보고 내부 통제** (ICFR: Internal Control over Financial Reporting)
- 재무제표에 영향을 미치는 통제
- 예: 급여 처리 서비스

#### SOC 2 ⭐
**보안, 가용성, 처리 무결성, 기밀성, 개인정보 보호** 통제
- 기술 기반 서비스 제공자
- 예: 클라우드, SaaS, 호스팅

#### SOC 3
SOC 2의 **공개 요약본**
- 세부 내용 없이 결과만
- 웹사이트 게시용

**이 Day는 SOC 2에 집중합니다.**

### 1.4 SOC 2 vs ISO 27001

| 구분 | SOC 2 | ISO 27001 |
|------|-------|-----------|
| 성격 | **감사 보고서** | 인증 |
| 발행 기관 | AICPA (미국) | ISO/IEC (국제) |
| 범위 | 서비스 제공자 | 모든 조직 |
| 평가 기준 | TSC (5개 영역) | 93개 통제항목 |
| 평가자 | CPA (공인회계사) | 인증 심사원 |
| 결과물 | SOC 2 보고서 | 인증서 |
| 공개 | **비공개** (고객에게만) | 공개 가능 |
| 유효 기간 | 보고서 기준일/기간 | 3년 |
| 인지도 | 미국 중심 | 전 세계 |

**핵심 차이:**
- **SOC 2**: 미국 시장 중심, 감사 보고서
- **ISO 27001**: 글로벌 표준, 인증서

### 1.5 누가 SOC 2를 필요로 하는가?

#### 서비스 제공자 (Service Organization)
- **SaaS 기업** (Salesforce, Slack, Zoom 등)
- **클라우드 서비스** (AWS, Azure, GCP)
- **호스팅/IDC**
- **데이터 센터**
- **결제 대행사** (PG)
- **보안 서비스**
- **HR/급여 아웃소싱**

#### 필요 이유
- **미국 고객** 요구
- **B2B SaaS** 고객 요구
- **입찰 조건**
- **투자 유치** (VC/PE가 요구)
- **고객 신뢰** 확보

## 2. Trust Services Criteria (TSC)

### 2.1 5개 Trust Services Categories

SOC 2는 **5개 영역**을 평가:
```
1. Security (보안) - 필수
2. Availability (가용성) - 선택
3. Processing Integrity (처리 무결성) - 선택
4. Confidentiality (기밀성) - 선택
5. Privacy (개인정보 보호) - 선택
```

**Security는 필수**, 나머지는 서비스에 따라 선택

### 2.2 CC 1.0: Security (보안) - 필수

**모든 SOC 2 보고서에 필수 포함**

#### 주요 통제 영역

**CC1: 통제 환경 (Control Environment)**
- 조직의 보안 문화
- 이사회 및 경영진의 감독
- 조직 구조 및 책임
- 역량 및 교육

**CC2: 커뮤니케이션 및 정보 (Communication and Information)**
- 내부 의사소통
- 외부 의사소통
- 정보 품질

**CC3: 위험 평가 (Risk Assessment)**
- 위험 식별
- 위험 분석
- 위험 대응

**CC4: 모니터링 활동 (Monitoring Activities)**
- 지속적 모니터링
- 평가
- 보고

**CC5: 통제 활동 (Control Activities)**
- 논리적 및 물리적 접근 통제
- 시스템 운영
- 변경 관리

**CC6: 논리적 및 물리적 접근 통제 (Logical and Physical Access Controls)**
- 사용자 인증
- 권한 관리
- 물리적 보안

**CC7: 시스템 운영 (System Operations)**
- 시스템 모니터링
- 백업 및 복구
- 용량 관리

**CC8: 변경 관리 (Change Management)**
- 변경 요청
- 변경 승인
- 변경 테스트
- 변경 배포

**CC9: 위험 완화 (Risk Mitigation)**
- 보안 사고 관리
- 취약점 관리
- 침입 탐지

### 2.3 A 1.0: Availability (가용성) - 선택

**시스템이 합의된 수준으로 사용 가능**

#### 주요 통제
- **가용성 목표** 설정 (SLA)
- **용량 관리**
- **모니터링**
- **백업 및 복구**
- **재해 복구 (DR)**
- **사고 대응**

**예시:**
- SaaS 업타임 99.9% 보장
- 데이터 백업 일일 1회
- 복구 시간 목표 (RTO) 4시간

### 2.4 PI 1.0: Processing Integrity (처리 무결성) - 선택

**시스템 처리가 완전하고, 유효하고, 정확하고, 시기적절하고, 승인됨**

#### 주요 통제
- **입력 검증**
- **처리 검증**
- **출력 검증**
- **오류 처리**
- **데이터 무결성**

**예시:**
- 결제 처리 시스템: 거래 금액 정확성
- 급여 처리: 급여 계산 정확성
- 송금 시스템: 이중 결제 방지

### 2.5 C 1.0: Confidentiality (기밀성) - 선택

**기밀로 지정된 정보를 보호**

#### 주요 통제
- **데이터 분류**
- **암호화** (저장, 전송)
- **접근 통제**
- **데이터 폐기**
- **NDA (비밀유지계약)**

**예시:**
- 고객 기밀 데이터 암호화
- 영업 비밀 접근 제한
- 퇴사자 데이터 접근 즉시 차단

### 2.6 P 1.0: Privacy (개인정보 보호) - 선택

**개인정보 수집, 사용, 보유, 공개, 폐기 시 개인정보 보호**

**GAPP (Generally Accepted Privacy Principles) 기반**

#### 주요 통제
- **동의** (Notice and Consent)
- **선택 및 동의** (Choice and Consent)
- **수집** (Collection)
- **사용, 보유 및 폐기** (Use, Retention, and Disposal)
- **접근** (Access)
- **공개** (Disclosure to Third Parties)
- **품질** (Quality)
- **모니터링 및 집행** (Monitoring and Enforcement)

**GDPR, CCPA 등과 유사**

## 3. SOC 2 보고서 유형

### 3.1 Type I vs Type II

#### Type I (특정 시점 평가)
**특정 날짜**에 통제가 **적절히 설계**되었는지 평가

**기간**: 특정 날짜 (예: 2024년 12월 31일)

**평가 내용:**
- 통제 **설계**의 적절성

**예시:**
- "2024년 12월 31일 기준, 통제가 적절히 설계됨"

**장점:**
- 빠름 (1~2개월)
- 비용 저렴

**단점:**
- 통제 **운영** 효과성 미확인

#### Type II (일정 기간 평가) ⭐ 일반적
**일정 기간** (최소 6개월) 동안 통제가 **효과적으로 운영**되었는지 평가

**기간**: 최소 6개월 (예: 2024년 7월 1일 ~ 12월 31일)

**평가 내용:**
- 통제 **설계**의 적절성
- 통제 **운영**의 효과성

**예시:**
- "2024년 7월 1일 ~ 12월 31일, 통제가 효과적으로 운영됨"

**장점:**
- 통제 실제 작동 확인
- 고객이 선호

**단점:**
- 시간 소요 (6개월 이상)
- 비용 높음

**대부분 Type II 선호**

### 3.2 보고서 구성

SOC 2 Type II 보고서는 일반적으로 **100~200 페이지**

#### Section I: 독립 감사인 보고서
CPA의 의견

#### Section II: 경영진 주장 (Management's Assertion)
서비스 조직 경영진의 통제 설명

#### Section III: 시스템 설명
- 서비스 개요
- 시스템 구성 요소
- 인프라
- 소프트웨어
- 인력
- 절차
- 데이터

#### Section IV: Trust Services Criteria
선택한 TSC (Security, Availability 등)

#### Section V: 통제 및 테스트 결과
- 각 통제 설명
- 감사인 테스트
- 테스트 결과
- **예외 사항** (발견된 문제)

**예외 사항 (Exceptions):**
- 통제가 효과적이지 않았던 경우
- 예: "백업 테스트가 10월에 누락됨"

**중요:** 예외가 있어도 보고서 발행 가능, 단 고객이 우려할 수 있음

## 4. SOC 2 준비 및 감사 절차

### 4.1 준비 단계 (6~12개월)

#### Phase 1: 스코핑 (Scoping) - 1개월
1. **범위 결정**
   - 어떤 서비스?
   - 어떤 시스템?
   - 어떤 위치?

2. **TSC 선택**
   - Security (필수)
   - Availability, PI, Confidentiality, Privacy (선택)

3. **보고서 유형 결정**
   - Type I vs Type II

**예시:**
```
범위: ABC SaaS 플랫폼
시스템: 웹 애플리케이션, 데이터베이스, AWS 인프라
위치: AWS us-east-1
TSC: Security + Availability
유형: Type II (6개월)
```

#### Phase 2: 레디니스 평가 (Readiness Assessment) - 2~3개월
1. **갭 분석**
   - 현재 통제 수준 vs TSC 요구사항

2. **개선 과제 도출**

3. **로드맵 수립**

#### Phase 3: 통제 구현 (Remediation) - 3~6개월
1. **정책 및 절차 수립**
   - 보안 정책
   - 접근 제어 정책
   - 변경 관리 절차
   - 사고 대응 절차

2. **기술적 통제 구현**
   - 접근 통제 (SSO, MFA)
   - 암호화
   - 로그 관리
   - 백업
   - 모니터링

3. **물리적 통제 구현**
   - 데이터센터 접근 통제
   - CCTV

4. **관리적 통제 구현**
   - 보안 교육
   - 배경 조사
   - 역할 및 책임

5. **증적 수집**
   - 정책 문서
   - 회의록
   - 교육 기록
   - 로그
   - 스크린샷

#### Phase 4: 사전 평가 (Pre-Assessment) - 선택 사항
비공식 모의 감사

#### Phase 5: 감사 준비 (Audit Readiness) - 1개월
1. **CPA 선정**
2. **감사 계획 수립**
3. **증적 정리**
4. **담당자 교육**

### 4.2 감사 단계 (2~3개월)

#### Phase 1: 계획 (Planning)
- 감사 범위 확인
- 일정 조율
- 샘플 선정 방법

#### Phase 2: 현장 작업 (Fieldwork)
**기간**: 2~4주

**활동:**
1. **문서 검토**
   - 정책, 절차
   - 시스템 문서

2. **인터뷰**
   - 경영진
   - IT 팀
   - 보안 팀
   - 인사 팀

3. **통제 테스트**
   - 샘플 검증
   - 예: 25개 사용자 계정 접근 권한 검토

4. **시스템 검사**
   - 설정 확인
   - 로그 검토

**샘플 크기:**
- 일반적으로 **25~40개 샘플**
- 자동화된 통제는 샘플 적음

#### Phase 3: 보고서 작성 (Reporting)
**기간**: 4~6주

1. **초안 작성**
2. **경영진 검토**
3. **예외 사항 논의**
4. **최종 보고서** 발행

### 4.3 감사 비용

#### 레디니스 평가
- $10,000 ~ $30,000

#### SOC 2 Type I
- $15,000 ~ $50,000

#### SOC 2 Type II
- $25,000 ~ $100,000+

**변수:**
- 조직 규모
- 시스템 복잡도
- TSC 개수
- 감사 회사

### 4.4 유지 관리

SOC 2는 **일회성이 아님**

#### 연간 갱신
대부분 **매년 갱신**
- 고객 요구
- 지속적 신뢰 유지

#### 지속적 컴플라이언스
- 통제 모니터링
- 증적 수집
- 정책 업데이트

## 5. 주요 통제 영역 상세

### 5.1 조직 및 관리 (Organization and Management)

#### 보안 조직
- CISO 또는 보안 책임자 지정
- 보안 팀 구성
- 역할 및 책임 명확화

#### 정책 및 절차
- 보안 정책
- 접근 제어 정책
- 데이터 분류 정책
- 사고 대응 계획

#### 위험 관리
- 연 1회 이상 위험 평가
- 위험 대응 계획

### 5.2 커뮤니케이션 (Communications)

#### 내부 커뮤니케이션
- 보안 정책 전파
- 정기 보안 회의

#### 외부 커뮤니케이션
- 고객에게 보안 정보 제공
- 사고 발생 시 고객 통지

### 5.3 위험 평가 및 모니터링

#### 위험 평가
- 연 1회 위험 평가
- 새로운 시스템·서비스 도입 시

#### 모니터링
- 보안 로그 모니터링
- SIEM
- 이상 징후 탐지

### 5.4 논리적 접근 통제

#### 사용자 계정 관리
- **고유 계정** 부여
- 공용 계정 금지

#### 인증
- **강력한 비밀번호** (최소 8자, 복잡성)
- **MFA** (다중 인증) - 특히 관리자 계정

#### 권한 관리
- **최소 권한 원칙**
- RBAC (역할 기반 접근 제어)

#### 정기 검토
- **분기 1회** 사용자 접근 권한 검토
- 불필요 권한 회수

#### 계정 프로비저닝/디프로비저닝
- 신규 입사자: 즉시 계정 생성
- 퇴사자: **즉시 계정 비활성화**

**테스트 예시:**
감사인이 25명의 퇴사자 샘플을 선정하여 계정이 당일 비활성화되었는지 확인

### 5.5 물리적 접근 통제

#### 데이터센터 보안
- **출입 통제** (카드, 생체인증)
- **방문자 관리** (동행, 로그)
- **CCTV** (24시간 녹화)

#### 클라우드 사용 시
- AWS, Azure, GCP의 **SOC 2 보고서** 활용
- "Carve-out" 또는 "Inclusive" 방법

**Carve-out:**
- 클라우드 제공자의 물리적 통제는 별도 (클라우드 제공자의 SOC 2 참조)

**Inclusive:**
- 클라우드 제공자의 통제를 보고서에 포함

### 5.6 시스템 운영

#### 변경 관리
- **변경 요청** 문서화
- **변경 승인** (CAB: Change Advisory Board)
- **변경 테스트**
- **변경 배포**
- **변경 로그** 유지

**테스트 예시:**
감사인이 25개 변경 건을 샘플링하여 승인·테스트 증적 확인

#### 백업
- **정기 백업** (일일, 주간)

## 6. 보안 컨설팅 관점의 시사점

### 6.1 SOC 2 컨설팅

#### 대상 고객
- 미국 진출 SaaS 기업
- B2B 클라우드 서비스
- 데이터 센터
- HR/급여 아웃소싱

#### 컨설팅 프로세스

**Phase 1: 레디니스 평가 (2~3개월)**
1. 스코핑
2. 갭 분석
3. 로드맵 수립

**Phase 2: 통제 구현 지원 (3~6개월)**
1. 정책·절차 작성
2. 기술적 통제 구현
3. 증적 수집 가이드
4. 모의 감사

**Phase 3: 감사 대응 지원 (2~3개월)**
1. CPA 선정 지원
2. 인터뷰 준비
3. 증적 제출 지원

**총 기간: 7~12개월**

### 6.2 SOC 2 vs ISO 27001 선택

| 상황 | 권장 |
|------|------|
| 미국 고객 중심 | SOC 2 |
| 글로벌 고객 | ISO 27001 |
| B2B SaaS | SOC 2 |
| VC 투자 유치 (미국) | SOC 2 |
| 입찰 (미국) | SOC 2 |
| 입찰 (유럽/아시아) | ISO 27001 |

**병행 인증:**
- 대기업은 **SOC 2 + ISO 27001** 모두 취득
- 중복 부분 많아 효율적

### 6.3 SOC 2 준비 팁

#### 1. 일찍 시작
최소 **1년 전** 준비 시작
- Type II는 6개월 운영 증적 필요

#### 2. 자동화
- IaC (Infrastructure as Code)
- 자동화된 접근 제어
- 자동화된 백업
→ 통제 일관성, 증적 수집 용이

#### 3. 증적 수집 체계화
- 문서 관리 시스템
- 스크린샷, 로그 자동 수집

#### 4. 스코프 최소화
초기에는 **핵심 서비스만**
- 범위가 넓으면 비용·시간 증가

#### 5. 경험 많은 CPA 선택
SOC 2 경험이 풍부한 감사 회사 선택

## 8. SOC 2 보고서 읽는 법 (고객 관점)

### 8.1 서비스 이용자가 확인할 사항

#### 1. 보고서 유형
- **Type II** 선호 (Type I은 운영 증적 없음)

#### 2. 보고서 기간
- 최소 **6개월** 이상
- 최근 **1년 이내** 보고서

#### 3. TSC
- **Security 필수**
- 서비스에 따라 Availability, PI 등 확인

#### 4. 예외 사항 (Exceptions)
- **예외가 많으면 문제**
- 예외의 심각성 평가

#### 5. 범위 (Scope)
- 내가 사용하는 **서비스가 포함**되었는지 확인

#### 6. 서브서비스 조직 (Subservice Organizations)
- 클라우드 제공자 (AWS, Azure 등)
- Carve-out인 경우 서브서비스 조직 SOC 2 별도 확인

### 8.2 레드 플래그 (Red Flags)

- Type I만 제공 (Type II 없음)
- 보고서가 **2년 이상 오래됨**
- **예외 사항 10개 이상**
- **중요 통제 누락** (예: MFA 없음)
- 범위가 **너무 좁음**
