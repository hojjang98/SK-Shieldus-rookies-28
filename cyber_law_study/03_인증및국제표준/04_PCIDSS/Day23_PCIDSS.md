# Day 23: PCI-DSS 카드 정보 보호 표준

## 1. PCI-DSS의 개요

### 1.1 PCI-DSS란?

**정식 명칭**: Payment Card Industry Data Security Standard (지불카드산업 데이터보안 표준)

**PCI-DSS**: 신용카드 정보를 **안전하게 처리·저장·전송**하기 위한 국제 보안 표준

**발행 기관**: PCI Security Standards Council (PCI SSC)
- Visa, Mastercard, American Express, Discover, JCB 등 주요 카드사가 공동 설립 (2006년)

### 1.2 제정 배경

- **카드 정보 유출 사고** 급증
- 카드사마다 상이한 보안 기준
- 통일된 **글로벌 표준** 필요
- 카드 회원 정보 보호 및 **사기 거래 방지**

### 1.3 PCI-DSS의 목적

1. **카드 소지자 데이터(CHD)** 보호
2. **카드 정보 유출** 방지
3. **보안 사고 감소**
4. **카드사 및 가맹점의 신뢰** 확보

### 1.4 적용 대상

PCI-DSS는 **카드 정보를 처리하는 모든 조직**에 적용:

#### (1) 가맹점 (Merchant)
카드 결제를 받는 사업자:
- 오프라인 매장
- 온라인 쇼핑몰
- 호텔, 레스토랑
- 주유소

#### (2) 서비스 제공자 (Service Provider)
카드 정보를 처리·저장·전송하는 사업자:
- PG사 (Payment Gateway)
- 결제 대행사
- 호스팅 업체
- 클라우드 서비스 제공자

### 1.5 PCI-DSS vs 기타 표준

| 구분 | PCI-DSS | ISO 27001 | ISMS-P |
|------|---------|-----------|--------|
| 범위 | **카드 정보만** | 정보보호 전반 | 정보보호+개인정보 |
| 적용 대상 | 카드 처리 조직 | 모든 조직 | 정보통신서비스 제공자 |
| 인증 의무 | 카드사 요구 시 | 자발적 | 일부 의무 |
| 발행 기관 | PCI SSC | ISO/IEC | 한국인터넷진흥원 |
| 유효 기간 | 1년 | 3년 | 3년 |
| 국제 인정 | ⭐⭐⭐ | ⭐⭐⭐ | ⭐ |

**특징:**
- PCI-DSS는 **카드 정보에 특화**된 표준
- **매년 재평가** 필요 (타 표준보다 엄격)

## 2. PCI-DSS의 구조

### 2.1 버전 연혁

- **2004년**: Visa, Mastercard 등 개별 보안 프로그램 운영
- **2006년**: PCI SSC 설립, PCI-DSS v1.0 발표
- **2010년**: PCI-DSS v2.0
- **2013년**: PCI-DSS v3.0
- **2018년**: PCI-DSS v3.2.1
- **2022년**: PCI-DSS v4.0 발표
- **2024년 3월**: v4.0 시행
- **2025년 3월**: v3.2.1 종료 (v4.0 완전 전환)

**현재 버전**: PCI-DSS v4.0

### 2.2 6대 목표 및 12개 요구사항

PCI-DSS는 **6개 목표**와 **12개 요구사항**으로 구성:
```
목표 1: 안전한 네트워크 및 시스템 구축 및 유지
  ├─ 요구사항 1: 네트워크 보안 통제 설치 및 유지
  └─ 요구사항 2: 모든 시스템 구성 요소에 안전한 구성 적용

목표 2: 계정 데이터 보호
  ├─ 요구사항 3: 저장된 계정 데이터 보호
  └─ 요구사항 4: 개방형 공용 네트워크를 통한 카드 소지자 데이터 전송 시 암호화

목표 3: 취약점 관리 프로그램 유지
  ├─ 요구사항 5: 악성 소프트웨어로부터 모든 시스템 및 네트워크 보호
  └─ 요구사항 6: 안전한 시스템 및 소프트웨어 개발 및 유지

목표 4: 강력한 접근 통제 조치 구현
  ├─ 요구사항 7: 비즈니스 필요에 따라 계정 데이터에 대한 접근 제한
  ├─ 요구사항 8: 사용자 식별 및 인증
  └─ 요구사항 9: 계정 데이터에 대한 물리적 접근 제한

목표 5: 네트워크 정기 모니터링 및 테스트
  ├─ 요구사항 10: 시스템 구성 요소 및 계정 데이터에 대한 모든 접근 로그 및 모니터링
  └─ 요구사항 11: 보안 시스템 및 네트워크 정기 테스트

목표 6: 정보보안 정책 유지
  └─ 요구사항 12: 모든 담당자에 대한 정보보안 정책 지원
```

### 2.3 주요 용어

#### (1) CHD (Cardholder Data) - 카드 소지자 데이터
카드 소지자를 식별할 수 있는 정보:
- **PAN** (Primary Account Number): 카드 번호
- **카드 소지자 이름**
- **서비스 코드**
- **유효 기간**

#### (2) SAD (Sensitive Authentication Data) - 민감 인증 데이터
카드 인증에 사용되는 민감 데이터:
- **CVV/CVC** (카드 뒷면 3자리 보안 코드)
- **PIN** (Personal Identification Number)
- **마그네틱 트랙 데이터** (Track 1, Track 2)

**중요:**
- SAD는 **승인 후 절대 저장 금지** (PCI-DSS 핵심 규칙)

#### (3) CDE (Cardholder Data Environment) - 카드 소지자 데이터 환경
CHD를 **저장·처리·전송**하는 시스템 및 네트워크 환경

## 3. 12개 요구사항 상세

### 3.1 요구사항 1: 네트워크 보안 통제 설치 및 유지

**목적:**
네트워크 경계에서 **무단 접근 차단**

#### 주요 통제
1. **방화벽 설치 및 구성**
   - CDE와 외부 네트워크 사이 방화벽
   - 인바운드/아웃바운드 트래픽 제한
   - 불필요한 포트 차단

2. **네트워크 다이어그램 유지**
   - CDE 범위 명확히 표시
   - 정기 업데이트

3. **DMZ 구성**
   - 외부 접속 서비스는 DMZ에 배치

**구현 예시:**
```
[인터넷]
    ↓
[방화벽 1] ← 외부 방화벽
    ↓
[DMZ] ← 웹 서버
    ↓
[방화벽 2] ← 내부 방화벽
    ↓
[CDE] ← 카드 정보 DB, 결제 시스템
```

### 3.2 요구사항 2: 모든 시스템 구성 요소에 안전한 구성 적용

**목적:**
시스템 **기본 설정 변경**, 불필요한 서비스 제거

#### 주요 통제
1. **기본 비밀번호 변경**
   - 벤더 기본 비밀번호 즉시 변경
   - 예: admin/admin → 강력한 비밀번호

2. **불필요한 서비스 비활성화**
   - 사용하지 않는 서비스, 프로토콜, 포트 제거

3. **구성 표준 수립**
   - 서버, 네트워크 장비 등 표준 설정

4. **암호화 프로토콜**
   - 강력한 암호화만 사용 (TLS 1.2 이상)
   - 약한 암호화 비활성화 (SSL, TLS 1.0/1.1)

**구현 예시:**
- SSH: 기본 포트 22 → 다른 포트로 변경
- 웹 서버: TLS 1.2 이상만 허용
- 데이터베이스: 불필요한 샘플 DB 삭제

### 3.3 요구사항 3: 저장된 계정 데이터 보호

**목적:**
저장된 카드 정보를 **암호화** 또는 **최소화**

#### 주요 통제
1. **데이터 최소화**
   - 필요한 CHD만 저장
   - 보유 기간 최소화

2. **SAD 저장 금지** ⭐ 가장 중요
   - CVV/CVC: **절대 저장 금지**
   - PIN: **절대 저장 금지**
   - 마그네틱 트랙 데이터: **승인 후 삭제**

3. **PAN 암호화**
   - 저장 시 **강력한 암호화** (AES-256)
   - 암호키는 PAN과 분리 보관

4. **PAN 마스킹**
   - 화면 표시 시 일부만 표시
   - 예: 1234-****-****-5678

**구현 예시:**
```
[올바른 저장]
PAN: 암호화된 값 (AES-256)
카드 소지자 이름: 홍길동
유효 기간: 12/25

[잘못된 저장 - 절대 금지!]
CVV: 123 ← 저장 금지!
PIN: 1234 ← 저장 금지!
```

### 3.4 요구사항 4: 개방형 공용 네트워크를 통한 카드 소지자 데이터 전송 시 암호화

**목적:**
전송 중 카드 정보 **도청 방지**

#### 주요 통제
1. **강력한 암호화**
   - HTTPS (TLS 1.2 이상)
   - 무선: WPA2 이상

2. **약한 암호화 금지**
   - SSL
   - TLS 1.0, 1.1
   - WEP

3. **신뢰할 수 있는 인증서**
   - 유효한 SSL 인증서 사용

**구현 예시:**
- 웹사이트: https:// (TLS 1.3)
- 무선 네트워크: WPA3
- VPN: IPSec 또는 TLS VPN

### 3.5 요구사항 5: 악성 소프트웨어로부터 모든 시스템 및 네트워크 보호

**목적:**
**악성코드 감염 방지**

#### 주요 통제
1. **백신 소프트웨어 설치**
   - CDE 내 모든 시스템

2. **정기 업데이트**
   - 백신 정의 파일 자동 업데이트

3. **정기 스캔**
   - 최소 주 1회 전체 스캔

4. **로그 유지**
   - 백신 로그 보관

**구현 예시:**
- 서버: 백신 설치 + 자동 업데이트
- PC: 엔드포인트 보안 솔루션
- 네트워크: 방화벽에서 악성코드 차단

### 3.6 요구사항 6: 안전한 시스템 및 소프트웨어 개발 및 유지

**목적:**
**보안 취약점 제거**

#### 주요 통제
1. **보안 패치 적용**
   - 중요 패치는 **30일 이내** 적용
   - 정기 패치 관리

2. **시큐어 코딩**
   - OWASP Top 10 대응
   - SQL Injection, XSS 등 방지

3. **웹 애플리케이션 방화벽 (WAF)**
   - 공개 웹 애플리케이션에 WAF 적용

4. **변경 관리**
   - 시스템 변경 시 보안 영향 평가

**구현 예시:**
```python
# 나쁜 예: SQL Injection 취약
query = "SELECT * FROM users WHERE id = '" + user_id + "'"

# 좋은 예: Prepared Statement
query = "SELECT * FROM users WHERE id = ?"
cursor.execute(query, (user_id,))
```

### 3.7 요구사항 7: 비즈니스 필요에 따라 계정 데이터에 대한 접근 제한

**목적:**
**최소 권한 부여**

#### 주요 통제
1. **역할 기반 접근 통제 (RBAC)**
   - 업무 역할에 따라 권한 차등

2. **기본 거부 원칙**
   - 명시적으로 허용하지 않으면 거부

3. **권한 정기 검토**
   - 분기 1회 이상

**구현 예시:**
- 영업팀: 주문 조회만 (PAN 마스킹)
- 결제팀: 결제 처리 권한
- IT팀: 시스템 관리 권한 (PAN 접근 최소화)

### 3.8 요구사항 8: 사용자 식별 및 인증

**목적:**
**사용자 신원 확인**

#### 주요 통제
1. **고유 계정**
   - 사용자별 개별 계정 부여
   - 공용 계정 금지

2. **강력한 비밀번호**
   - 최소 8자 (v4.0: 12자 권장)
   - 복잡성: 영문+숫자+특수문자
   - 90일마다 변경

3. **다중 인증 (MFA)**
   - CDE 접근 시 MFA 필수
   - 비밀번호 + OTP

4. **계정 잠금**
   - 로그인 6회 실패 시 잠금

**구현 예시:**
```
[비밀번호 정책]
- 최소 12자 이상
- 영문 대소문자 + 숫자 + 특수문자
- 90일마다 변경
- 이전 4개 비밀번호 재사용 금지
- 6회 로그인 실패 시 30분 잠금

[MFA]
- CDE 접근: 비밀번호 + Google Authenticator
```

### 3.9 요구사항 9: 계정 데이터에 대한 물리적 접근 제한

**목적:**
**물리적 침입 방지**

#### 주요 통제
1. **출입 통제**
   - CDE 서버실 출입카드
   - 방문자 동행

2. **CCTV**
   - 서버실 24시간 녹화
   - 90일 보관

3. **미디어 보호**
   - 백업 테이프 등 안전한 장소 보관
   - 폐기 시 물리적 파쇄

4. **장비 보호**
   - POS 단말기 변조 방지

**구현 예시:**
- 서버실: 생체인증 + 출입카드
- 백업: 별도 잠금 캐비닛
- 하드디스크 폐기: 물리적 파쇄 또는 소각

### 3.10 요구사항 10: 시스템 구성 요소 및 계정 데이터에 대한 모든 접근 로그 및 모니터링

**목적:**
**이상 행위 탐지**

#### 주요 통제
1. **로그 기록**
   - 모든 CDE 접근 기록
   - 로그인/로그아웃
   - 관리자 작업
   - 카드 정보 접근

2. **로그 보존**
   - 최소 **1년** 보관
   - 최근 **3개월**은 즉시 분석 가능

3. **로그 보호**
   - 위·변조 방지
   - 별도 로그 서버

4. **일일 로그 검토**
   - 자동화된 로그 모니터링
   - 이상 징후 알림

**구현 예시:**
- SIEM 도입 (Splunk, ELK 등)
- 로그 중앙 집중화
- 실시간 알림 (비정상 접근, 대량 다운로드 등)

### 3.11 요구사항 11: 보안 시스템 및 네트워크 정기 테스트

**목적:**
**취약점 조기 발견**

#### 주요 통제
1. **취약점 스캔**
   - **분기 1회** 외부 스캔 (ASV 승인 업체)
   - **분기 1회** 내부 스캔

2. **침투 테스트**
   - **연 1회** 이상
   - 중요 변경 후

3. **무선 스캔**
   - 분기 1회 무선 AP 탐지

4. **파일 무결성 모니터링 (FIM)**
   - 중요 파일 변경 탐지

**구현 예시:**
- 외부 스캔: ASV (Approved Scanning Vendor) 업체 활용
- 침투 테스트: 외부 모의해킹 업체
- FIM: Tripwire, OSSEC 등

### 3.12 요구사항 12: 모든 담당자에 대한 정보보안 정책 지원

**목적:**
**보안 인식 제고**

#### 주요 통제
1. **정보보안 정책 수립**
   - PCI-DSS 준수 정책
   - 연 1회 검토

2. **위험 평가**
   - 연 1회 이상 위험 평가

3. **보안 교육**
   - 채용 시 교육
   - 연 1회 이상 정기 교육

4. **인시던트 대응**
   - 침해사고 대응 계획
   - 연 1회 이상 훈련

5. **서비스 제공자 관리**
   - PCI-DSS 준수 계약
   - 정기 점검

**구현 예시:**
```
[보안 교육 프로그램]
- 신규 입사자: 즉시 PCI-DSS 교육 (1시간)
- 전 직원: 연 1회 온라인 교육
- 개발자: 시큐어 코딩 교육 (연 1회)
- 침해사고 대응 훈련: 연 1회
```

## 4. PCI-DSS 준수 레벨

### 4.1 가맹점 레벨 (Merchant Levels)

**연간 거래량**에 따라 4개 레벨:

| 레벨 | 연간 거래량 | 평가 방법 |
|------|-----------|----------|
| Level 1 | 600만 건 이상 | QSA 현장 심사 (ROC) |
| Level 2 | 100만~600만 건 | SAQ + 분기별 취약점 스캔 |
| Level 3 | 2만~100만 건 | SAQ + 분기별 취약점 스캔 |
| Level 4 | 2만 건 미만 | SAQ (카드사 재량) |

### 4.2 서비스 제공자 레벨 (Service Provider Levels)

| 레벨 | 연간 거래량 | 평가 방법 |
|------|-----------|----------|
| Level 1 | 30만 건 이상 | QSA 현장 심사 (ROC) |
| Level 2 | 30만 건 미만 | SAQ 또는 ROC |

### 4.3 평가 방법

#### (1) SAQ (Self-Assessment Questionnaire) - 자가 평가
소규모 가맹점이 **스스로 평가**

**SAQ 유형:**
- **SAQ A**: 전자상거래 (제3자 결제 대행)
- **SAQ A-EP**: 전자상거래 (자체 결제 페이지)
- **SAQ B**: POS 단말기만
- **SAQ C**: 웹 기반 POS
- **SAQ D**: 기타 모든 가맹점

#### (2) ROC (Report on Compliance) - 현장 심사
대규모 가맹점은 **QSA가 현장 심사**

**QSA (Qualified Security Assessor)**: PCI SSC 인증 심사원

#### (3) ASV Scan - 취약점 스캔
**분기 1회** 외부 취약점 스캔

**ASV (Approved Scanning Vendor)**: PCI SSC 승인 스캔 업체

## 5. PCI-DSS 준수 절차

### 5.1 준비 단계 (3~6개월)

#### Phase 1: 범위 결정 (1개월)
1. **CDE 범위 식별**
   - 카드 정보를 처리·저장·전송하는 시스템
   - 네트워크 다이어그램 작성

2. **적용 레벨 확인**
   - 연간 거래량 확인

3. **적용 SAQ 유형 결정**

#### Phase 2: 갭 분석 (1~2개월)
1. **현황 진단**
   - 12개 요구사항 대비 현재 수준

2. **부족한 부분 파악**
   - 우선순위 결정

#### Phase 3: 개선 (3~6개월)
1. **기술적 통제 구현**
   - 방화벽, 암호화, 백신 등

2. **관리적 통제 구현**
   - 정책, 절차, 교육

3. **물리적 통제 구현**
   - 출입 통제, CCTV

### 5.2 평가 단계 (1~2개월)

#### SAQ 작성 또는 QSA 심사
- Level 1, 2: QSA 현장 심사
- Level 3, 4: SAQ 자가 평가

#### ASV 취약점 스캔
- 분기 1회

### 5.3 제출 및 인증

#### AoC (Attestation of Compliance) 제출
카드사에 준수 확인서 제출

**유효 기간**: 1년

### 5.4 유지 관리

- **분기별**: ASV 스캔
- **연간**: 재평가 (SAQ 또는 ROC)
- **수시**: 중요 변경 시 재평가

## 6. 보안 컨설팅 관점의 시사점

### 6.1 PCI-DSS 준수 컨설팅

#### 대상 고객
- 온라인 쇼핑몰
- 오프라인 매장 (POS)
- PG사
- 호스팅 업체

#### 컨설팅 프로세스

**Phase 1: 범위 및 현황 진단 (1개월)**
1. CDE 범위 결정
2. 네트워크 다이어그램 작성
3. 갭 분석

**Phase 2: 개선 계획 수립 (1개월)**
1. 우선순위 결정
2. 로드맵 수립
3. 비용 산정

**Phase 3: 통제 구현 (3~6개월)**
1. 네트워크 분리 (방화벽, DMZ)
2. 암호화 (DB, 전송)
3. 접근 통제
4. 로그 관리
5. 정책·절차 수립

**Phase 4: 평가 준비 (1개월)**
1. 내부 점검
2. 증적 준비
3. 모의 심사

**Phase 5: 평가 (1~2개월)**
1. ASV 스캔
2. QSA 심사 또는 SAQ 작성

**총 기간: 6~12개월**

### 6.2 PCI-DSS 범위 최소화 전략

**핵심**: CDE 범위를 최소화하면 **비용·노력 절감**

#### 전략 1: 카드 정보 저장 최소화
- 필요한 경우에만 저장
- 가능하면 토큰화 (Tokenization)

#### 전략 2: 네트워크 분리
- CDE를 별도 네트워크로 격리
- 방화벽으로 엄격히 통제

#### 전략 3: 제3자 결제 활용
- PG사에 결제 위임 (SAQ A)
- 자체 CDE 구축 불필요

**예시:**
```
[범위 최소화 전]
모든 서버가 CDE → 모든 서버에 PCI-DSS 적용 → 비용 ↑

[범위 최소화 후]
결제 서버만 CDE → 결제 서버만 PCI-DSS 적용 → 비용 ↓
```

### 6.3 토큰화 (Tokenization)

**토큰화**: 실제 PAN을 **무작위 토큰**으로 대체

**장점:**
- PAN 저장 불필요 → **CDE 범위 축소**
- 유출되어도 무의미 (토큰은 무작위 값)

**예시:**
```
[원본 PAN]
1234-5678-9012-3456

[토큰]
TOK-A1B2C3D4E5F6
```

**활용:**
- 토큰만 저장
- 실제 PAN은 토큰 서비스 업체가 보관
- 결제 시 토큰 → PAN 변환

## 7. 실무 사례

### 사례 1: 온라인 쇼핑몰 (Level 3)

**상황:**
D쇼핑몰 (연간 50만 건)

**요구사항:**
- Level 3 → SAQ D + ASV 스캔

**PCI-DSS 준수:**
1. **네트워크 분리**
   - 결제 서버를 별도 VLAN
   - 방화벽으로 격리

2. **암호화**
   - PAN 저장 시 AES-256
   - HTTPS (TLS 1.3)

3. **CVV 저장 금지**
   - 승인 후 즉시 삭제

4. **로그 관리**
   - 1년 보관
   - 일일 모니터링

5. **ASV 스캔**
   - 분기 1회

**결과:**
- SAQ D 작성 완료
- ASV 스캔 통과
- 카드사 제출

### 사례 2: PG사 (Level 1 Service Provider)

**상황:**
EPG사 (연간 1,000만 건)

**요구사항:**
- Level 1 → QSA 현장 심사 (ROC)

**PCI-DSS 준수:**
1. **CDE 물리적 분리**
   - 전용 데이터센터
   - 생체인증 출입 통제

2. **HSM (Hardware Security Module)**
   - 암호키 안전 보관

3. **24시간 모니터링**
   - SOC (Security Operations Center) 운영

4. **침투 테스트**
   - 연 2회 (필수 연 1회 초과)

5. **QSA 심사**
   - 1년마다 현장 심사

**결과:**
- ROC 통과
- PCI-DSS 인증서 발급

### 사례 3: 토큰화 도입

**상황:**
F쇼핑몰, PCI-DSS 비용 부담

**기존 방식:**
- PAN 자체 저장
- 전체 시스템이 CDE
- PCI-DSS 준수 비용 연간 5억원

**토큰화 도입:**
1. 토큰 서비스 업체 선정
2. PAN → 토큰 변환
3. 토큰만 저장

**결과:**
- CDE 범위 90% 축소
- PCI-DSS 준수 비용 연간 1억원
- 보안 수준 향상

## 8. PCI-DSS v4.0 주요 변경 사항 (2024년)

### 8.1 주요 변화

#### (1) 비밀번호 정책 강화
- **최소 길이**: 8자 → **12자 권장**

#### (2) 다중 인증 (MFA) 확대
- CDE 접근 시 **모든 계정**에 MFA 필수

#### (3) 맞춤형 구현 (Customized Implementation)
- 일부 요구사항에 대해 **자체 방법** 허용
- 단, 동등 이상의 보안 수준 입증 필요

#### (4) 지속적 모니터링 강조
- 연 1회 평가 → **지속적 준수** 문화

#### (5) 새로운 요구사항 추가
- 멀티 테넌트 서비스 제공자 보안
- 암호화 알고리즘 정기 검토

### 8.2 전환 일정
- **2024년 3월**: v4.0 시행
- **2025년 3월**: v3.2.1 종료 (완전 전환)
