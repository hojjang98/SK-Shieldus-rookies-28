# Day 23: PCI-DSS (결제 카드 산업 데이터 보안 표준)

## 1. PCI-DSS의 개요

### 1.1 PCI-DSS란?

**정식 명칭**: Payment Card Industry Data Security Standard (결제 카드 산업 데이터 보안 표준)

**PCI-DSS**: 신용카드 정보 유출 및 사기 범죄를 예방하기 위해 글로벌 카드사들이 공동으로 제정한 **카드 소유자 데이터 보호를 위한 글로벌 보안 기술 및 운영 표준**

**관리 기관**: **PCI SSC** (Payment Card Industry Security Standards Council)

* 5대 글로벌 카드사(Visa, MasterCard, American Express, Discover, JCB)가 공동 설립

### 1.2 제정 배경

* 전자상거래 발달로 인한 **신용카드 정보 유출 사고 급증**
* 과거 각 카드사별로 상이했던 보안 규정(예: Visa의 CISP, MasterCard의 SDP 등)의 파편화
* 가맹점과 서비스 제공자의 혼란을 줄이고 **통합된 글로벌 보안 표준** 마련 필요성 대두 (2004년 최초 제정)
* **현재 버전**: **PCI-DSS v4.0** (2024년 3월 31일부로 v3.2.1 은퇴, v4.0 전면 적용)

### 1.3 PCI-DSS의 목적

조직이 **카드 소유자 데이터(CHD)**와 **민감한 인증 데이터(SAD)**를 안전하게 보호하기 위해:

1. 안전한 결제 환경(네트워크 및 시스템) 구축
2. 신용카드 정보의 안전한 저장 및 전송
3. 강력한 접근 통제 및 지속적인 모니터링 체계 확립

### 1.4 적용 대상

카드 소유자 데이터를 **저장(Store), 처리(Process), 전송(Transmit)**하는 모든 주체:

1. **가맹점 (Merchants)**: 온/오프라인 쇼핑몰, 마트, 호텔 등
2. **서비스 제공자 (Service Providers)**: PG사(결제대행), VAN사, 호스팅 및 클라우드 제공자
3. **매입사 (Acquirers)** 및 **발급사 (Issuers)**

## 2. 핵심 용어 및 보호 대상 데이터

### 2.1 CDE (Cardholder Data Environment)

**카드 소유자 데이터 환경**: 카드 데이터를 저장, 처리, 전송하는 모든 시스템과 이에 연결된 네트워크 환경. **PCI-DSS 심사의 대상 범위(Scope)**가 됨.

### 2.2 보호 대상 데이터: CHD vs SAD

PCI-DSS는 결제 데이터를 크게 두 가지로 분류하며, 취급 기준이 매우 엄격함.

| 구분 | 포함 항목 | 저장 허용 여부 | 암호화 필요 여부 |
| --- | --- | --- | --- |
| **CHD** (카드 소유자 데이터) | **PAN (Primary Account Number, 카드번호)** | ⭕ (저장 가능) | ⭐⭐⭐ (필수) |
|  | 카드 소유자 이름 | ⭕ | 선택 (PAN과 함께 저장 시 보호) |
|  | 만료 일자 (Expiration Date) | ⭕ | 선택 |
|  | 서비스 코드 (Service Code) | ⭕ | 선택 |
| **SAD** (민감한 인증 데이터) | Full Track Data (마그네틱 스트라이프/칩 데이터) | ❌ **(승인 후 절대 저장 불가)** | - |
|  | CAV2/CVC2/CVV2/CID (카드 뒷면 3~4자리 보안코드) | ❌ **(승인 후 절대 저장 불가)** | - |
|  | PIN / PIN Block (비밀번호) | ❌ **(승인 후 절대 저장 불가)** | - |

**핵심 규칙:** **SAD(민감한 인증 데이터)는 결제 승인 완료 후 그 어떤 이유로도 저장해서는 안 됨.**

## 3. PCI-DSS 6대 목표와 12개 주요 요구사항 (v4.0 기준)

PCI-DSS는 매우 구체적이고 기술적인 지침을 제공합니다. 6개의 큰 목표 아래 12개의 주요 요구사항으로 구성됩니다.

### 목표 1: 안전한 네트워크 및 시스템 구축 및 유지

* **요구사항 1**: CDE를 보호하기 위해 네트워크 보안 제어(방화벽 등)를 설치하고 유지 관리
* **요구사항 2**: 모든 시스템 구성 요소에 안전한 구성(Secure Configuration) 적용 (벤더 기본값/비밀번호 사용 금지)

### 목표 2: 계정 데이터 보호

* **요구사항 3**: 저장된 계정 데이터 보호 (PAN 마스킹, 암호화, 해싱 등)
* **요구사항 4**: 개방된 공중 네트워크(인터넷 등)를 통해 전송되는 카드 소유자 데이터 암호화 (TLS 등)

### 목표 3: 취약점 관리 프로그램 유지

* **요구사항 5**: 모든 시스템을 악성 소프트웨어로부터 보호하고 백신 프로그램의 주기적 업데이트 유지
* **요구사항 6**: 안전한 시스템 및 소프트웨어 개발 및 유지 (보안 패치, 시큐어 코딩 적용)

### 목표 4: 강력한 접근 통제 조치 구현

* **요구사항 7**: 업무상 알 필요(Need-to-Know)에 따라 시스템 구성 요소 및 카드 소유자 데이터에 대한 접근 제한
* **요구사항 8**: 시스템 구성 요소에 대한 접근 식별 및 인증 (모든 사용자에게 고유 ID 부여, MFA 다중 인증 필수)
* **요구사항 9**: 카드 소유자 데이터에 대한 물리적 접근 제한 (통제구역 설정, 방문자 기록)

### 목표 5: 네트워크 정기 모니터링 및 테스트

* **요구사항 10**: 시스템 구성 요소 및 카드 소유자 데이터에 대한 모든 접근 로깅 및 모니터링
* **요구사항 11**: 시스템 및 네트워크 보안의 정기적 테스트 (ASV 취약점 스캔, 모의해킹 등)

### 목표 6: 정보 보안 정책 유지

* **요구사항 12**: 정보 보안을 지원하는 조직적 정책 및 프로그램 유지 (위험 평가, 보안 인식 교육, 사고 대응 계획)

## 4. 인증 레벨 및 검증 방식

카드 연간 거래 건수에 따라 레벨(Level)을 나누며, 검증 방식이 다릅니다. (Visa 기준)

| 레벨 | 연간 거래 건수 | 검증 방식 | 네트워크 스캔 |
| --- | --- | --- | --- |
| **Level 1** | **600만 건 이상** | **RoC** (QSA에 의한 현장 심사) | 분기별 ASV 스캔 |
| **Level 2** | 100만 건 ~ 600만 건 | **SAQ** (자가 진단) 또는 RoC | 분기별 ASV 스캔 |
| **Level 3** | 2만 건 ~ 100만 건 (e-커머스) | **SAQ** (자가 진단) | 분기별 ASV 스캔 |
| **Level 4** | 2만 건 미만 | **SAQ** (자가 진단) | 분기별 ASV 스캔 |

### 주요 심사 도구

1. **RoC (Report on Compliance)**: 공인 보안 평가사(**QSA**, Qualified Security Assessor)가 직접 현장에 방문하여 수행하는 엄격한 심사 보고서.
2. **SAQ (Self-Assessment Questionnaire)**: 조직 스스로 요구사항 준수 여부를 평가하는 자가 진단표. (결제 환경에 따라 A, A-EP, D 등 다양한 유형 존재)
3. **ASV (Approved Scanning Vendor)**: PCI SSC에서 승인한 외부 스캐닝 업체를 통해 분기별 1회 외부망 취약점 스캔 필수 수행.

## 5. 보안 컨설팅 관점의 시사점

### 5.1 범위 축소(Scope Reduction)가 핵심

PCI-DSS 컨설팅의 제1원칙은 **"평가 범위(CDE)를 최소화하라"**입니다. 범위가 넓을수록 막대한 보안 비용과 심사 리소스가 소모됩니다.

* **네트워크 분리 (Network Segmentation)**: 방화벽을 통해 CDE 네트워크를 일반 업무망과 완벽히 격리.
* **토큰화 (Tokenization)**: 실제 카드번호(PAN) 대신 무의미한 토큰 값을 내부 시스템에 저장하여 CDE 범위를 대폭 축소.

### 5.2 매우 구체적이고 기술적인 표준 (Prescriptive)

ISO 27001이 '관리 체계'를 묻는다면(예: "비밀번호 정책이 있는가?"), PCI-DSS는 '구체적인 수치'를 묻습니다(예: "비밀번호는 최소 12자리 이상, 영문/숫자 혼합, 90일마다 변경하는가?"). 따라서 기술적 보안 솔루션 도입이 거의 필수적입니다.

### 5.3 ISO 27001 / ISMS-P와의 비교

| 구분 | PCI-DSS | ISO 27001 | ISMS-P |
| --- | --- | --- | --- |
| **보호 대상** | 카드 결제 데이터 (CHD, SAD) | 기업의 모든 정보 자산 | 정보 자산 + 개인정보 |
| **요구 수준** | **매우 구체적/기술적 (Pass/Fail)** | 관리적 중심, 유연함 | 관리적 + 기술적 (한국 법규) |
| **심사 주체** | QSA (공인 평가사) | 인증 기관 (BSI, SGS 등) | KISA 및 지정 심사기관 |

## 6. 실무 사례

### 사례 1: 글로벌 e-커머스 플랫폼 (범위 축소의 정석)

**D커머스:**

* 연간 카드 결제 1,000만 건 이상 (Level 1 대상)
* 자체 DB에 카드번호(PAN)를 암호화하여 저장 중이었음.

**문제점:**

* 웹서버, WAS, DB, 내부망 전체가 CDE(평가 범위)로 지정되어 보안 인프라 투자 비용이 수십억 단위로 예상됨.

**컨설팅 해결책 (Tokenization 도입):**

1. 결제 정보 저장을 외부 전문 PG사(또는 토큰화 서비스 제공자)로 완전 이관.
2. D커머스 내부 DB에는 실제 카드번호 대신 **'결제 토큰(Token)'**만 저장.
3. 결과적으로 내부망 대부분이 PCI-DSS 평가 범위에서 제외됨.
4. SAQ-A(가장 간단한 양식) 수준으로 범위가 축소되어 비용과 시간을 혁신적으로 절감.

### 사례 2: 결제 대행사 (PG사) / Service Provider

**E페이 (PG사):**

* 가맹점을 대신해 카드사와 직접 통신 및 결제 처리.

**역할 및 과제:**

* PCI-DSS Level 1 **Service Provider** 자격 필수 (미인증 시 글로벌 카드사 망 연동 불가).
* v3.2.1에서 v4.0으로의 전환 심사 준비.

**PCI-DSS v4.0 대응:**

1. **타겟 위험 분석(Targeted Risk Analysis)** 수행: 보안 통제 주기를 유연하게 가져가기 위해 조직 자체적인 위험 분석 체계 구축.
2. **인증 강화**: CDE에 접속하는 모든 계정에 MFA(다중 인증) 전면 적용.
3. **스크립트 모니터링**: 결제 페이지에 삽입되는 타사 스크립트(Magecart 공격 방어) 무결성 모니터링 솔루션 도입.
4. **결과**: QSA 심사를 거쳐 RoC 획득, 글로벌 파트너십 유지.

## 7. 체크리스트

### 결제 데이터 보호

* [ ] 신용카드 결제 승인 후 SAD(CVC, PIN 등)를 저장하고 있지 않은가?
* [ ] 데이터베이스에 저장된 PAN(카드번호)은 강력하게 암호화(또는 마스킹)되어 있는가? (마스킹 시 첫 6자리, 마지막 4자리만 표시)
* [ ] 결제 데이터 전송 구간에 TLS v1.2 이상의 안전한 암호화 통신이 적용되어 있는가?

### 시스템 및 접근 통제

* [ ] CDE(결제 환경) 네트워크가 방화벽/VLAN 등으로 다른 망과 완벽히 격리되어 있는가?
* [ ] 시스템, 라우터 등의 벤더사 기본 제공 비밀번호를 모두 변경했는가?
* [ ] CDE에 접속하는 모든 관리자/사용자에게 MFA(다중 인증)를 적용했는가?

### 모니터링 및 테스트

* [ ] CDE 시스템의 모든 접속 및 감사 로그를 중앙 집중화하여 모니터링하고 있는가?
* [ ] 분기별 1회 ASV(공인 외부 스캐닝 업체)를 통한 취약점 스캔을 수행하는가?
* [ ] 연 1회 이상 CDE 환경에 대한 내/외부 모의해킹(Penetration Test)을 수행하는가?

## 8. 학습 정리

오늘 학습한 핵심 내용:

* PCI-DSS는 카드 결제 데이터(CHD, SAD) 보호를 위한 글로벌 보안 표준 (현재 v4.0).
* CVC, PIN과 같은 **SAD는 결제 승인 후 절대 저장 불가**.
* 6대 목표와 12개의 주요 요구사항으로 구성되며, 매우 구체적이고 기술적인 지침을 제공.
* 인증 레벨은 거래 건수에 따라 나뉘며, Level 1은 QSA의 현장 심사(RoC) 필수.
* 실무 적용의 핵심은 네트워크 분리나 토큰화를 통한 **"CDE(평가 범위) 최소화"**.
